\section{Design}
\label{sec:design}

This section presents the \hadl type system and the gradual validation mechanism that unifies type generation, workflow generation, and policy generation. We describe the abstract syntax, the correspondence between \hadl types and JavaScript types, and the three forms of gradual validation that check LLM-produced types, programs, and authorization policies at runtime. \sect{sec:semantics} gives the operational semantics and soundness theorem.

\subsection{Syntax}
\label{sec:design-syntax}

Figure~\ref{fig:syntax} presents the abstract syntax. A program is pragmas followed by declarations. Types include base types, \texttt{Schema}, arrow types, \texttt{Policy}, and named references. Expressions add to standard imperative constructs the four primitives, \texttt{js} interop, \texttt{eval}, \texttt{enforce}, and comparisons. The \texttt{as} clause on \texttt{gen}/\texttt{agent} accepts a string literal or a variable for dynamic principal resolution.

\hadl's base types correspond to JavaScript types ($\texttt{String}{\to}\texttt{string}$, etc.), and all runtime values are represented as JSON. The type checker uses inference with subsumption. \texttt{gen} and \texttt{js} are oracular, requiring explicit annotations at binding sites. Three types play a special role in gradual validation: \texttt{Schema} is the \emph{type of types}, arrow types are the types of workflows, and \texttt{Policy} is the type of authorization rules. All three share the property that their concrete values are generated by the LLM at runtime.

\subsection{Gradual Validation}
\label{sec:design-gradual}

The three gradual-validation annotations (\texttt{Schema}, arrow types, \texttt{Policy}) share the same mechanism: static checking treats them as universally compatible (inspired by the unknown type in gradual typing~\cite{siek2006gradual}), and rather than inserting casts, \hadl re-invokes the type checker over downstream code when a concrete value materializes.

\paragraph{Gradual type generation.} A \texttt{gen} annotated with \texttt{Schema} produces a JSON Schema at runtime. Since the concrete type does not exist at compile time, the subtyping relation treats \texttt{Schema} as universally compatible ($T \le \texttt{Schema}$ and $\texttt{Schema} \le T$ for all $T$). When a schema materializes, the interpreter converts it via a mapping that sends \texttt{null} to \texttt{()}, \texttt{boolean} to \texttt{Bool}, \texttt{integer}/\texttt{number} to \texttt{Number}, \texttt{string}/\texttt{enum} to \texttt{String}, and compound schemas (\texttt{array}/\texttt{object}/\texttt{\$ref}) back to \texttt{Schema}, then registers the result in the type environment and re-invokes the type checker over the remaining program. A \texttt{Schema}-typed variable can then appear in annotation position (e.g.~\texttt{let booking: form\_schema = gen "..."}). Re-checking after materialization verifies downstream bindings. If it fails and \texttt{@retries} is active, the runtime re-invokes the causal gen call with the type error as feedback.

\paragraph{Gradual workflow generation.} A \texttt{gen} annotated with an arrow type (e.g.~\texttt{(String -> Number)}) instructs the LLM to produce a complete \hadl workflow as structured output. The entire AST is serializable, so the LLM can generate a JSON-serialized \texttt{Program}. \texttt{eval} deserializes, validates, and executes this value within the current interpreter context with closure-like semantics. This enables hierarchical plan generation where an outer workflow delegates sub-task planning to the LLM.

\paragraph{Gradual policy generation.} A \texttt{gen} annotated with \texttt{Policy} instructs the LLM to produce a Cedar policy as JSON. Gradual policy generation separates policy \emph{generation} from \emph{activation}: a \texttt{Policy}-annotated binding stores the policy without activating it, and the explicit \texttt{enforce} expression installs it. Because \texttt{enforce} is an ordinary expression, it can appear inside conditional branches, enabling human-in-the-loop approval. Installation enforces (1) a deny-all guard (rejecting blanket \texttt{forbid}), (2) action-type validation delegated to the Cedar parser, and (3) append-only merging, so restrictions accumulate monotonically. Scoped principal validation across \texttt{as} clauses is not an install-time property. It is enforced at invocation time by the Cedar check in $\rulename{Oracle-Success}$. If a generated policy fails to compile, feedback-driven self-healing re-invokes the causal gen call with the Cedar error.

\subsection{Threat Model and Limitations}
\label{sec:design-threat}

\hadl trusts the programmer-authored text, the type checker, the runtime re-checker, and the Cedar decision procedure and treats every LLM output as untrusted. The retry budget $\MRetry$ bounds how many untrusted attempts the runtime consumes per site. Validation rejects outputs that fail structural checks of the annotated type, and every \texttt{gen}/\texttt{agent} is Cedar-checked against the installed policy before the LLM is called. A denial triggers retry. Four threat classes remain outside this model: \emph{prompt injection} against the LLM via composed \texttt{ask} input, \emph{semantic misalignment} of generated policies (the deny-all guard catches pathological policies but not merely over-permissive ones), \emph{data exfiltration} via \texttt{say}, and \emph{side effects} of \texttt{js} blocks executing under ambient host authority. \sect{app:threat} expands on these. The formal results are stated relative to an \emph{eventually-truthful} oracle (\defn{def:oracle-truthful}). An adversarial or permanently-lying oracle exhausts the budget and the configuration fails closed by absence of transition.

\label{sec:design-limitations}
The type system is deliberately simple: monomorphic, with homogeneous arrays, declared-field record checking (no row polymorphism or width subtyping), no sum/union/intersection types, \texttt{Schema} as top during static checking, and oracular \texttt{js} blocks. Relaxing these is not required for gradual validation to be sound. \sect{app:limitations} enumerates each restriction.
