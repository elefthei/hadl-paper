\section{Design}
\label{sec:design}

This section presents the \hadl type system and the gradual validation mechanism that unifies type generation, workflow generation, and policy generation. We describe the abstract syntax, the correspondence between \hadl types and JavaScript types, and the three forms of gradual validation that check LLM-produced types, programs, and authorization policies at runtime. \sect{sec:semantics} gives the operational semantics and soundness theorem.

\subsection{Syntax}
\label{sec:design-syntax}

Figure~\ref{fig:syntax} presents the abstract syntax. A program is pragmas followed by declarations. Types include base types (now including \texttt{Principal} and \texttt{Path}), \texttt{Schema}, arrow types, \texttt{Policy}, and named references. Expressions add to standard imperative constructs the four primitives, \texttt{js} interop, \texttt{eval}, \texttt{enforce}, comparisons, and the principal introducers \texttt{root} and \texttt{restrict}\;$e$. The \texttt{as} clause on \texttt{gen}/\texttt{agent} takes an expression of type \texttt{Principal}; the legacy literal-string form is dropped in favor of named principals (\texttt{let user: Principal = restrict root}).

\hadl's base types correspond to JavaScript types ($\texttt{String}{\to}\texttt{string}$, etc.), and all runtime values are represented as JSON. The type checker uses inference with subsumption. \texttt{gen} and \texttt{js} are oracular, requiring explicit annotations at binding sites. Three types play a special role in gradual validation: \texttt{Schema} is the \emph{type of types}, arrow types are the types of workflows, and \texttt{Policy} is the type of authorization rules. All three share the property that their concrete values are generated by the LLM at runtime.

\subsection{Gradual Validation}
\label{sec:design-gradual}

The three gradual-validation annotations (\texttt{Schema}, arrow types, \texttt{Policy}) share the same mechanism: static checking treats them as universally compatible (inspired by the unknown type in gradual typing~\cite{siek2006gradual}), and rather than inserting casts, \hadl re-invokes the type checker over downstream code when a concrete value materializes.

\paragraph{Gradual type generation.} A \texttt{gen} annotated with \texttt{Schema} produces a JSON Schema at runtime. The subtyping relation treats \texttt{Schema} as universally compatible ($T \le \texttt{Schema}$ and $\texttt{Schema} \le T$ for all $T$). When a schema materializes, the interpreter converts it to the corresponding \hadl{} type, registers it, and re-invokes the type checker over the remainder of the program; on failure (within \texttt{@retries}) the runtime feeds the type error back to the causal \texttt{gen}. See Appendix~\ref{app:gradual-detail} for the JSON-Schema mapping and re-checking discipline.

\paragraph{Gradual workflow generation.} A \texttt{gen} annotated with an arrow type instructs the LLM to produce a complete \hadl{} workflow as structured output. \texttt{eval} deserializes, validates, and executes this value within the current interpreter context. Hierarchical plan generation lets an outer workflow delegate sub-task planning to the LLM.

\paragraph{Gradual policy generation.} A \texttt{gen} annotated with \texttt{Policy} produces a Cedar policy as JSON. \texttt{enforce} is a separate expression that installs the policy --- separating generation from activation enables conditional or human-in-the-loop approval. Installation enforces a deny-all guard, action-type validation, and append-only merging, so restrictions accumulate monotonically. Scoped principal validation across \texttt{as} clauses is enforced at invocation time by the Cedar check in $\rulename{Oracle-Success}$. A failed compile triggers feedback-driven self-healing.

\subsection{Threat Model and Limitations}
\label{sec:design-threat}

\hadl{} trusts the programmer-authored text, the type checker, the runtime re-checker, and the Cedar decision procedure, and treats every LLM output as untrusted. The retry budget $\MRetry$ bounds how many untrusted attempts the runtime consumes per site. Validation rejects outputs that fail structural checks of the annotated type, and every \texttt{gen}/\texttt{agent} is Cedar-checked before the LLM is called; denial triggers retry. Four threat classes remain outside this model --- prompt injection, semantic misalignment of generated policies, \texttt{say}-based exfiltration, and host-side \texttt{js} effects --- expanded in \sect{app:threat}. Formal results are stated relative to an \emph{eventually-truthful} oracle (\defn{def:oracle-truthful}); an adversarial or permanently-lying oracle exhausts the budget and the configuration fails closed.

\label{sec:design-limitations}
The type system is deliberately simple: monomorphic, with homogeneous arrays, declared-field record checking (no row polymorphism or width subtyping), no sum/union/intersection types, \texttt{Schema} as top during static checking, and oracular \texttt{js} blocks. Relaxing these is not required for gradual validation to be sound. \sect{app:limitations} enumerates each restriction.
