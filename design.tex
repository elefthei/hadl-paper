\section{Design}
\label{sec:design}

This section presents the HADL syntax and the gradual validation mechanism that unifies the checking of LLM-generated types, workflows, and policies under a single pattern. We describe the abstract syntax, the correspondence between HADL types and JavaScript types, and the three forms of gradual validation that check types, workflows, and authorization policies produced by LLMs against their declared types as the workflow runs.

\subsection{Syntax}
\label{sec:design-syntax}

Figure~\ref{fig:syntax} presents the abstract syntax of HADL. A program consists of pragmas followed by declarations. Types include base types, \texttt{Schema} (the type of types), workflow types, \texttt{Policy} for authorization rules, array types, record types, and named references. Expressions include standard imperative constructs, the four domain-specific primitives (\texttt{ask}, \texttt{say}, \texttt{gen}, \texttt{agent}), JavaScript interop (\texttt{js}), workflow invocation (\texttt{eval}), policy activation (\texttt{enforce}), and comparison operators. The \texttt{as} clause on \texttt{gen} and \texttt{agent} accepts either a string literal or a variable reference for dynamic principal resolution.

\input{figures/syntax}

HADL's base types correspond to JavaScript types, and all values are represented as JSON. \texttt{gen} and \texttt{js} expressions are treated as oracles that can produce any type, requiring explicit type annotations at binding sites. \texttt{ask} and \texttt{agent} always return \texttt{String}, and \texttt{say} returns~$()$. Workflow types have the form $(T_1 \to T_2)$, and \texttt{Policy} is the type of authorization rules. All three special annotations, \texttt{Schema}, workflow, and \texttt{Policy}, share the property that their concrete values are produced by the LLM and validated as the program executes.

\subsection{Gradual Validation}
\label{sec:design-gradual}

HADL addresses the dynamic nature of LLM outputs through \emph{gradual validation}. Three annotations, \texttt{Schema}, workflow types, and \texttt{Policy}, signal that the value will be produced by the LLM and checked against its declared type the moment it is put to use, inspired by the unknown type in gradual typing~\cite{siek2006gradual}. When such a value materializes, the runtime validates it, installs it into the execution context, and enforces it over the remainder of the program. All three forms share provenance-driven self-healing that retries the causal gen call if validation fails.

\subsubsection{Gradual Type Validation}
\label{sec:design-gradual-types}

A \texttt{gen} call annotated with \texttt{Schema} produces a JSON Schema value that specifies fields, their types, and their constraints. Because the concrete type is not yet known when the binding is first reached, the subtyping relation treats \texttt{Schema} as universally compatible, meaning both $T \le \texttt{Schema}$ and $\texttt{Schema} \le T$ hold for all types $T$. This lets \texttt{Schema}-annotated bindings pass the check without committing to a concrete type until the LLM delivers one. When the schema materializes, the interpreter converts it to a concrete JS type, registers it in the type environment, and re-invokes the type checker over the region from this binding to the next schema generation (or the end of the program) before execution resumes.

The schema value can then appear in type annotation position for subsequent bindings, as in \texttt{let booking: form\_schema = gen "..."}. When a schema-typed variable is mutated via assignment, the same re-checking process runs. If re-checking fails and \texttt{@retries} is active, provenance tracking identifies the causal gen call and re-invokes it with the type error as feedback, up to the declared retry limit. Compound JSON Schema types remain as \texttt{Schema}, and field-level property access is performed by the JavaScript engine on the underlying JSON values.

\subsubsection{Gradual Workflow Validation}
\label{sec:design-gradual-workflows}

A \texttt{gen} call annotated with a workflow type such as \texttt{(String -> Number)} instructs the LLM to produce a complete HADL workflow as structured output. The entire HADL AST is serializable, so the LLM can generate a JSON-serialized \texttt{Program} containing a workflow declaration with a typed parameter and a body. The \texttt{eval} expression deserializes this value, validates it, and executes it within the current interpreter context with closure-like semantics. At the type level, if the workflow has type $(A \to R)$, the argument is checked against $A$ and the result has type $R$. If the type is \texttt{Schema}, checking is deferred until the concrete type arrives. This enables hierarchical plan generation where an outer workflow delegates subtask planning to the LLM, and the generated workflow is a typed program validated at the call site.

\subsubsection{Gradual Policy Validation}
\label{sec:design-gradual-policies}

A \texttt{gen} call annotated with \texttt{Policy} instructs the LLM to produce a Cedar authorization policy as a JSON value. Gradual policy validation separates policy \emph{generation} from policy \emph{activation}. A \texttt{Policy}-annotated binding stores the generated policy without activating it, and the explicit \texttt{enforce} expression installs it. Because \texttt{enforce} is an ordinary expression, it can appear inside conditional branches, enabling human-in-the-loop approval. The type checker verifies that \texttt{enforce} targets a \texttt{Policy}-typed variable, and the expression returns~$()$.

Installation enforces three invariants. First, a \emph{deny-all guard} rejects any blanket \texttt{forbid} rule that would block all operations. Second, policies are merged \emph{append-only}, so new rules can tighten access but cannot remove existing rules, and Cedar's \texttt{forbid}-over-\texttt{permit} precedence guarantees that restrictions accumulate monotonically across multiple \texttt{enforce} expressions. Third, \emph{scoped principal validation} checks that all \texttt{as} clause references between the current \texttt{enforce} and the next one name principals defined in the active policy, catching mismatches early. If the generated policy is invalid, the same provenance-driven self-healing mechanism used for schema errors re-invokes the causal gen call with the Cedar error and valid schema as feedback.
