\section{Design}
\label{sec:design}

This section presents the HADL type system and the gradual generation mechanism that unifies type generation, workflow generation, and policy generation under a single pattern. We describe the abstract syntax, the correspondence between HADL types and JavaScript types, and the three forms of gradual generation that allow LLMs to produce types, programs, and authorization policies at runtime.

\subsection{Syntax}
\label{sec:design-syntax}

Figure~\ref{fig:syntax} presents the abstract syntax of HADL. A program consists of pragmas followed by declarations. Types include base types, \texttt{Schema} (the type of types), arrow types for workflows, \texttt{Policy} for authorization rules, and named references. Expressions include standard imperative constructs, the four domain-specific primitives (\texttt{ask}, \texttt{say}, \texttt{tool}, \texttt{agent}), JavaScript interop (\texttt{js}), workflow invocation (\texttt{eval}), policy activation (\texttt{enforce}), and comparison operators. The \texttt{as} clause on \texttt{tool} and \texttt{agent} expressions accepts either a string literal or a variable reference for dynamic principal resolution.

\begin{figure}[t]
\[\begin{array}{rcll}
P & ::= & \pi^*\; d^* & \textit{program} \\[4pt]
\pi & ::= & \texttt{@policy}\; s & \textit{policy pragma} \\
    & \mid & \texttt{@retries}\; n & \textit{retry pragma} \\[4pt]
d & ::= & \texttt{workflow}\; f(\overline{x_i{:}T_i}) \to T\; \{e\} & \textit{workflow declaration} \\
  & \mid & \texttt{type}\; X = T & \textit{type declaration} \\[4pt]
T & ::= & () \mid \texttt{String} \mid \texttt{Number} \mid \texttt{Bool} & \textit{base types} \\
  & \mid & \texttt{Schema} & \textit{type of types} \\
  & \mid & \texttt{Policy} & \textit{authorization policy} \\
  & \mid & (T_1 \to T_2) & \textit{arrow type} \\
  & \mid & \{ \overline{x_i{:}T_i} \} & \textit{record type} \\
  & \mid & X & \textit{named type reference} \\[4pt]
e & ::= & s \mid n \mid b & \textit{literals} \\
  & \mid & x & \textit{variable reference} \\
  & \mid & \texttt{let}\; x {:} T = e \mid \texttt{let}\; x = e & \textit{immutable binding} \\

  & \mid & \texttt{say}\; s & \textit{human output} \\
  & \mid & \texttt{tool}\; s\; [\texttt{as}\; (s \mid x)] & \textit{LLM tool call} \\

HADL's base types correspond to JavaScript types (\texttt{String}$\to$\texttt{string}, \texttt{Number}$\to$\texttt{number}, \texttt{Bool}$\to$\texttt{boolean}, $()$$\to$\texttt{null}), and all runtime values are represented as JSON. The type checker uses type inference with subsumption. \texttt{tool} and \texttt{js} expressions are treated as oracles that can produce any type, requiring explicit type annotations at binding sites. Three types play a special role in gradual generation: \texttt{Schema} is the \emph{type of types}, arrow types $(T_1 \to T_2)$ are the types of workflows, and \texttt{Policy} is the type of authorization rules. All three share the property that their concrete values are generated by the LLM at runtime.

\subsection{Gradual Generation}
\label{sec:design-gradual}

HADL addresses the inherently dynamic nature of LLM outputs through \emph{gradual generation}, in which three type annotations (\texttt{Schema}, arrow types, and \texttt{Policy}) signal that the annotated value will be generated by the LLM at runtime. \texttt{Schema} is treated as universally compatible during static checking, inspired by the unknown type in gradual typing~\cite{siek2006gradual}, but rather than inserting casts at boundaries, HADL re-invokes the type checker over downstream code when concrete types materialize. When such a value materializes, the runtime validates it, installs it into the execution context, and enforces it over the remainder of the program. All three forms share provenance-driven self-healing that retries the causal tool call if validation fails.

\subsubsection{Gradual Type Generation}
\label{sec:design-gradual-types}

A \texttt{tool} call annotated with \texttt{Schema} produces a JSON Schema value at runtime, providing a structured specification of fields, their types, and their constraints. Because the concrete type does not exist at compile time, the subtyping relation treats \texttt{Schema} as universally compatible, meaning both $T \le \texttt{Schema}$ and $\texttt{Schema} \le T$ hold for all types $T$. This allows \texttt{Schema}-annotated bindings to pass static checking while deferring actual type verification to runtime. When a schema materializes, the interpreter converts it to a concrete HADL type using the mapping $\sigma$ defined in Figure~\ref{fig:schema-to-type}, registers it in the type environment, and re-invokes the type checker over the remaining program before execution resumes.

\begin{figure}[t]
\[
\sigma(\tau) = \begin{cases}
() & \text{if } \tau = \texttt{null} \\
\texttt{Bool} & \text{if } \tau = \texttt{boolean} \\
\texttt{Number} & \text{if } \tau = \texttt{integer} \text{ or } \tau = \texttt{number} \\
\texttt{String} & \text{if } \tau = \texttt{string} \text{ or } \tau = \texttt{enum} \\
\texttt{Schema} & \text{if } \tau = \texttt{array} \text{ or } \tau = \texttt{object} \text{ or } \tau = \texttt{\$ref}
\end{cases}
\]
\caption{The schema-to-type mapping $\sigma$. Converts a runtime JSON Schema type $\tau$ into a HADL type. Primitive schemas map to concrete HADL types. Compound schemas (arrays, objects, references) remain as \texttt{Schema}.}
\label{fig:schema-to-type}
\end{figure}

The schema value can then appear in type annotation position for subsequent bindings. For example, in the binding \texttt{let booking: form\_schema = tool "..."}, the variable \texttt{form\_schema} holds a schema value whose materialized type governs the type of \texttt{booking}. When such a binding is evaluated, the interpreter deserializes the LLM's response as a JSON Schema, converts it to a concrete HADL type via $\sigma$, registers the type in the environment so that the variable can be used in type annotation positions, and re-invokes the type checker over the remaining expressions. When a schema-typed variable is mutated via assignment, the same process re-parses and re-checks. If re-checking fails and the \texttt{@retries} pragma is active, provenance tracking identifies the causal tool call and re-invokes it with the type error as feedback, up to the declared retry limit.

The staged mechanism provides two guarantees. First, at compile time, all statically known types are verified before any LLM call is made. Second, at runtime, the type checker is re-invoked with the concrete schema type substituted into the environment, verifying that downstream bindings are consistent with the materialized type. Compound JSON Schema types map to \texttt{Schema} under $\sigma$, so field-level property access is performed by the JavaScript engine at runtime on the underlying JSON values.

\subsubsection{Gradual Workflow Generation}
\label{sec:design-gradual-workflows}

A \texttt{tool} call annotated with an arrow type such as \texttt{(String -> Number)} instructs the LLM to produce a complete HADL workflow as structured output. The entire HADL AST is serializable, so the LLM can generate a JSON-serialized \texttt{Program} containing a workflow declaration with typed parameters and a body. The \texttt{eval} expression deserializes this value, validates it, and executes it within the current interpreter context with closure-like semantics. At the type level, if the function type is an arrow $(A \to R)$, arguments are checked against $A$ and the result type is $R$; if the type is \texttt{Schema}, checking is deferred to runtime. This enables hierarchical plan generation where an outer workflow delegates sub-task planning to the LLM, and the generated workflow is a full typed program that the type checker can verify at the call site.

\subsubsection{Gradual Policy Generation}
\label{sec:design-gradual-policies}

A \texttt{tool} call annotated with \texttt{Policy} instructs the LLM to produce a Cedar authorization policy as a JSON value at runtime. Gradual policy generation separates policy \emph{generation} from policy \emph{activation}: a \texttt{Policy}-annotated binding stores the generated policy without activating it, and the explicit \texttt{enforce} expression installs it in the runtime. Because \texttt{enforce} is an ordinary expression, it can appear inside conditional branches, enabling human-in-the-loop approval. The type checker verifies at compile time that \texttt{enforce} targets a \texttt{Policy}-typed variable, and the expression returns~$()$.

Installation enforces four invariants. First, a \emph{deny-all guard} rejects any blanket \texttt{forbid} rule that would block all operations. Second, \emph{action type validation} rejects unknown action types before they reach Cedar. Third, policies are merged \emph{append-only}: new rules can tighten access but cannot remove existing rules, and Cedar's \texttt{forbid}-over-\texttt{permit} precedence guarantees that restrictions accumulate monotonically across multiple \texttt{enforce} expressions. Fourth, \emph{scoped principal validation} checks that all \texttt{as} clause references between the current \texttt{enforce} and the next one name principals defined in the active policy, catching mismatches early. The \texttt{as} clause accepts both string literals and variable references for dynamic principal resolution. If the generated policy fails to compile, the same provenance-driven self-healing mechanism used for schema errors re-invokes the causal tool call with the Cedar error and valid schema as feedback.
