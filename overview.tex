\section{Overview}
\label{sec:overview}

We introduce \pact through two self-contained examples. The 
clinical-trial workflow of Figure~\ref{fig:clinical-trial} covers 
gradual validation of types and policies. A second example, the 
car-recommendation workflow of Figure~\ref{fig:overview-workflow}, 
covers workflow generation. Together they illustrate all three 
forms of gradual validation and self-healing that \pact provides.

\pact programs consist of workflow declarations, which are typed 
functions that compose four domain-specific primitives with 
standard imperative constructs. \texttt{ask} prompts a human 
operator and returns a textual answer (\texttt{String}), \texttt{say} prints a 
string and returns unit. \texttt{gen} sends an LLM prompt to an 
LLM session configured to return structured data and requires an 
explicit type annotation at the binding site. \texttt{agent} sends 
an instruction to an multi-round LLM agent and does not require a structured data response. These are first-class 
expressions in the language, composing with \texttt{let} and 
\texttt{var} bindings, \texttt{if}, \texttt{while}, \texttt{for}, 
and standard arithmetic and field projections.

\paragraph{Types and policies (Figure~\ref{fig:clinical-trial}).}
Line~5 binds identifier \texttt{crf} with a \texttt{Schema} annotation, a special
\pact type indicating the type-of-types expressed as a JSON schema description (draft 2020-12~\cite{jsonschema2020}).
The LLM delivers the schema, the interpreter resolves it and 
re-checks the remainder of the program against the concrete shape. 
If any \texttt{crf}-typed record has a field accessed later, such as \texttt{visit.cost} or 
\texttt{visit.patient\_id}, forward syntactic field-access 
analysis over the let-gen continuation triggers a validation step.
On validation failure, the runtime appends a 
missing-field error event and retries the same \texttt{gen} call---we detail
self-healing in the next paragraph. Note, the 
analysis is sound but over-approximate, meaning a projection in a 
never-taken branch still forces the oracle to supply that field. 
Refining to a path-sensitive analysis is future work. Line~6 introduces
the \texttt{coordinator} principal via
\texttt{restrict root}. Lines~7--8 produce a blinding policy that
line~9 activates via
\texttt{enforce}, after which every \texttt{gen} and \texttt{agent}
call must declare a principal in $\mathrm{dom}(\MPol)$ and every
file or directory operation the agent then performs is checked
just-in-time by the runtime monitor (\sect{sec:implementation}). The
\texttt{for}
loop on line~15 iterates patient IDs. Inside the loop,
\texttt{visit} is bound with the \texttt{crf} type annotation so
every LLM output is validated against the concrete schema before
the field accesses on lines~16--17 execute. The visit \texttt{gen} on
line~13 declares principal \texttt{coordinator} via an explicit
\texttt{as} clause; other LLM calls run under the default
principal \texttt{root}. 
\texttt{@retries 3} on line~1 activates the feedback-driven 
self-healing budget (\sect{sec:design-gradual}), so that on any 
validation failure the runtime re-invokes the causal \texttt{gen} 
with the violated constraint appended to the original prompt.


\paragraph{Self-healing in detail.}
It is worth tracing the self-healing episode in 
Figure~\ref{fig:clinical-trial} step by step, because the causal 
nature of the repair is what distinguishes \pact from 
assertion-based approaches. When the schema \texttt{gen} on 
line~5 returns, the runtime does not immediately proceed to 
generate patient visits. Instead it performs a forward syntactic 
analysis over the continuation of the schema-producing binding, 
collecting every field projection that the bound variable 
\texttt{crf} participates in downstream. In the clinical-trial 
workflow this analysis finds \texttt{visit.cost} and 
\texttt{visit.patient\_id} in the body of the \texttt{for} loop. 
If the generated schema does not provide those fields, the runtime 
constructs a feedback string describing the missing fields and 
retries the schema \texttt{gen} before any visit data is collected 
at all.

This is the key difference from a workflow that checks fields only 
at the point of use. In an assertion-based system, the missing 
field is not discovered until a visit record is actually generated 
and inspected. At that point the schema defect is repaired by 
retrying the visit generator with the schema injected into its 
prompt, but the same defect will be rediscovered independently for 
every subsequent visit. If the workflow generates three patient 
visits, the same structural error is caught three separate times, 
and three separate retries consume tokens. \pact repairs the 
schema once at the source, before any dependent generation has 
run, and all subsequent visit generations conform on their first 
attempt. This also does not require adding assertions, simply
use LLM generated values as if they are correctly typed, and let the runtime
self-heal against the context.

\paragraph{Workflow generation (Figure~\ref{fig:overview-workflow}).}
Agentic workflows often delegate subtasks whose structure is 
itself the subject of the current reasoning step. \pact makes 
workflows first-class by allowing a \texttt{gen} call to be 
annotated with an arrow type. In the car-recommendation workflow 
of Figure~\ref{fig:overview-workflow}(a), lines~3--4 generate a 
schema describing a single car, and lines~5--6 ask the LLM to 
produce a scorer with type \texttt{(car\_schema -> Number)}, where 
the parameter type is itself a schema generated at runtime. The 
LLM responds with a complete \pact workflow serialized as a JSON 
AST. The \texttt{eval} call on line~13 deserializes that workflow 
and executes it in the caller's environment. Before execution 
begins, the type checker verifies that the argument supplied to 
the scorer has the type \texttt{car\_schema} and that the workflow 
promises to return a \texttt{Number}. A mismatch at either 
position triggers the arrow-type heal rule, which retries the 
workflow-generating \texttt{gen}, not the caller.

The generated workflow appears in Figure~\ref{fig:overview-workflow}(b). 
It takes a single car record, calls \texttt{gen} three times to 
obtain numeric ratings for comfort, cost, and safety, and averages 
them via an arithmetic expression. The outer workflow calls this 
scorer three times inside a \texttt{while} loop, updating the 
running best score and best car name on each iteration. The 
important point is that the outer workflow never specifies how 
scoring works. It only declares the type the scorer must satisfy. 
The LLM fills in the scoring logic, and \pact enforces the 
declared type contract at the point where that logic is first put 
to use.
