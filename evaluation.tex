\section{Evaluation}
\label{sec:evaluation}

We evaluate \hadl on the clinical-trial workflow of Figure~\ref{fig:clinical-trial}.
The goal is to test the two central mechanisms, gradual schema typing and per-invocation Cedar
authorization, against baselines that must implement the same properties by hand.

\subsection{Research Questions and Method}
\label{sec:eval-rqs}

\textbf{RQ1 (schema validation):} Does the \texttt{crf} type annotation catch visit data that
does not conform to the generated schema, and does the language prevent this from propagating?
\textbf{RQ2 (authorization):} Does the Cedar policy block an unauthorized principal while allowing
an authorized one to complete the workflow?
\textbf{RQ3 (baselines):} How much code and manual enforcement is required to replicate the same
safety properties in LangGraph and DSPy?

We run all five systems $K=3$ times on the clinical-trial workflow driven by a fixed scripted
transcript.
\hadl and all baselines use the same LLM backend.
Each baseline uses the identical prompt text as \hadl:
the schema \texttt{gen} prompt, the policy \texttt{gen} prompt, the patient-ID prompt, and for
each patient visit the same bare visit prompt \emph{without} injecting the schema.
The two \emph{plain} baselines (LangGraph, DSPy) access \texttt{visit["cost"]} and
\texttt{visit["patient\_id"]} directly with no validation or authorization.
The two \emph{assertion} baselines (LangGraph+assertions, DSPy+assertions) add Pydantic
field-presence checks and a manual \texttt{check\_authorization()} call at each call site,
retrying with schema-injected prompts on failure.

\paragraph{Plain baselines fail.}
LangGraph and DSPy without assertions fail on every run.
The bare visit prompt \texttt{"Generate a visit for pid in trial"} produces a medically correct CRF
record (fields such as \texttt{tender\_joint\_count} and \texttt{das28\_crp}) without a
\texttt{cost} field, because no constraint communicated to the LLM that \texttt{cost} would be
accessed later.
The direct field access \texttt{visit["cost"]} raises a \texttt{KeyError} and the workflow crashes.
Three missed violations per system ($K=3$ runs) reflect that the frameworks provide no mechanism
to detect or prevent the omission.

\paragraph{Pact completes with automated self-healing.}
\hadl completed all three runs successfully.
On each run, the initial \texttt{crf} schema generation omitted the \texttt{cost} field, a
tracking field not surfaced by a bare clinical prompt.
\hadl's field-access scanner detected that \texttt{visit.cost} is referenced in the
for-loop body before any visit call is made, found \texttt{cost} absent from the generated schema,
and automatically retried the schema generation with the type error as feedback.
The healed schema included \texttt{cost} and all subsequent visit calls conformed on the first
attempt.
This self-healing event is counted once per run (\texttt{caught}$=3$ across $K=3$ runs,
\texttt{missed}$=0$), detected and repaired structurally with no developer intervention.
The \texttt{as "coordinator"} annotation fires Cedar unconditionally before each visit call and
\texttt{as "safety\_board"} fires before the agent call.
No authorization violation occurred because both principals are permitted by the generated policy.

\paragraph{Assertion baselines succeed but at significant cost.}
LangGraph+assertions and DSPy+assertions both complete but need 10 queries per run instead of 7
(three extra retry calls, one per patient visit).
On each first attempt the bare visit prompt omits \texttt{cost}. Pydantic catches the missing field
(\texttt{caught}$=9$ across three runs), and the retry re-issues the call with the schema and error
message injected manually.
The additional queries more than double the runtime (81\,s vs.\ 38\,s) and increase output tokens
nearly threefold (4605 vs.\ 1732) as the LLM re-generates each visit.
Authorization is enforced by a \texttt{check\_authorization()} call written by the developer.
Omitting it from any call site would silently bypass the check.
The \hadl program that achieves the same guarantees is 14 lines while the assertion baselines require
128--154 lines.

\subsection{Threats to Validity}
\label{sec:eval-threats}

This is a focused case study, not a broad benchmark: one workflow, one model family. Schema-healing triggered on every run --- the bare clinical prompt consistently omitted \texttt{cost} and \hadl{} repaired it on first retry --- but reliability depends on context; the structural guarantee, not LLM behavior, is the claim. The authorization comparison is partly qualitative: assertion baselines \emph{can} be made correct manually, at the cost of 114--140 added lines, doubled runtime, and reliance on developer diligence.
