\section{Threat Model and Out-of-Scope Concerns}
\label{app:threat}

\paragraph{Trust boundaries.} \pact treats the programmer-authored program text,
the static type checker, the runtime re-checker, and the installed Cedar
decision procedure as trusted. It treats every LLM output, whether a value, a
schema, a workflow AST, or a policy, as untrusted. The retry budget $\MRetry$
declared by \texttt{@retries} is a bound on how many untrusted attempts the
runtime will consume before failing closed on a given generation site.

\paragraph{What is caught.} Validation rejects an LLM output that fails the
structural checks of the annotated type: a schema whose $\sigma$-image is not
representable, a workflow AST that fails to type-check at its use site, or a
generated Cedar policy that fails the install discipline (effect must be
\texttt{forbid} for every rule, and no rule may target \texttt{root}). Every
\texttt{gen} and \texttt{agent} invocation carries the static premise
$\PrinDeclared{\MPol}{\MPrin}$ --- the principal named by the surrounding
\texttt{as} clause (or $\texttt{root}$ by default) is declared in the current
policy state. The action and resource of any specific filesystem access are
checked just-in-time by the runtime monitor of \sect{sec:implementation}; a deny
blocks the operation and fails closed. The two install premises together
guarantee that no \texttt{enforce} step can relax an already-installed
restriction or revoke \texttt{root}'s authority.

\begin{sloppypar}
\paragraph{What is not caught.} The runtime monitor's Cedar requests use
\texttt{Action} $\in \{\texttt{Read}, \texttt{Write}, \texttt{Exec}\}$ and
\texttt{Resource} ranging over files and directories the agent actually touches;
wildcards \texttt{*} are supported. Authorization decisions therefore depend on
which file an agent ultimately reads or writes, not on the call site of the
\texttt{gen}/\texttt{agent} expression. Four categories of threat remain outside
the model. First, \emph{prompt injection} against the LLM itself: a \texttt{gen}
call whose prompt is composed with untrusted \texttt{ask} input can still be
steered by an adversary who controls the user, even though every actual
filesystem access is policy-checked. Second, \emph{semantic misalignment} of
generated policies: the install discipline filters out rules that are
syntactically out of bounds, but an LLM-generated policy that is merely
\emph{over-permissive} with respect to operator intent will install and appear
well-formed. Third, \emph{data exfiltration} via \texttt{say}, where an
already-authorized workflow emits sensitive values to the human. Fourth,
\emph{side-effecting host-language expressions}, which execute arbitrary
JavaScript under the host runtime's ambient authority. These gaps are out of
scope for gradual validation as we define it; the install discipline prevents
policy relaxation but cannot recover from a malicious or inappropriate
\emph{initial} policy.  \end{sloppypar}

\section{Type-System Limitations}
\label{app:limitations}

\subsection{Gradual-validation details}
\label{app:gradual-detail}

\paragraph{Schema-to-type mapping.} When a \texttt{Schema} value materializes,
the interpreter applies a mapping $\sigma$ that sends \texttt{null} to
\texttt{()}, \texttt{boolean} to \texttt{Bool}, \texttt{integer}/\texttt{number}
to \texttt{Number}, \texttt{string}/\texttt{enum} to \texttt{String}, and
compound schemas (\texttt{array}/\texttt{object}/\texttt{\$ref}) back to
\texttt{Schema}, then registers the result in the type environment so a
\texttt{Schema}-typed variable may appear in annotation position (e.g.\
\texttt{let booking: form\_schema = gen "..."}).

\paragraph{Re-checking discipline.} Re-checking after materialization verifies
downstream bindings against the resolved type. If a downstream field access or
annotation fails to type-check and \texttt{@retries} is active, the runtime
re-invokes the causal \texttt{gen} call with the type error as feedback. The
same loop applies to workflow and policy generation: a continuation that fails
to type-check, an \texttt{eval} site whose argument types disagree with the
materialized arrow, or a Cedar parse failure each appends a feedback event and
re-invokes the causal \texttt{gen}.

\paragraph{Workflow generation.} The entire \pact{} AST is serializable, so an
arrow-typed \texttt{gen} produces a JSON-serialized \texttt{Program}.
\texttt{eval} deserializes, validates argument and return types against the
workflow type, and executes the result with closure-like semantics in the
caller's environment.

\paragraph{Policy generation.} A \texttt{Policy}-annotated binding stores the
policy without activating it; the explicit \texttt{enforce} expression installs
it, and because \texttt{enforce} is an ordinary expression it can appear inside
conditional branches (enabling human-in-the-loop approval).

\subsection{Restrictions of the type system}

The type system deliberately makes several simplifications to keep gradual
validation tractable.

\begin{itemize} \item \textbf{Arrays are homogeneously element-typed.}
\texttt{array[T]} requires every element to have type $T$; there are no tuple
types and no heterogeneous arrays.  \item \textbf{Records check declared fields
only.} A record type $\{\overline{x_i{:}T_i}\}$ requires every listed field to
be present with the listed type, but the runtime value may carry additional
fields. There is no row polymorphism, no width subtyping, and no structural
subtyping between record types.  \item \textbf{No unions or intersections.}
There are no sum types, union types, or refinement types. Values whose shape
depends on runtime choice must flow through \texttt{Schema} and be re-checked at
use.  \item \textbf{\texttt{Schema} defers structural checking.} \texttt{Schema}
is the type of types: a \texttt{Schema}-typed binding carries no structural
information during static checking. Field-level validity is deferred to the
post-materialization re-checker, which resolves the concrete shape and re-checks
the continuation. For compound schemas that $\sigma$ maps back to
\texttt{Schema}, field access is resolved at runtime by the host engine.  \item
\textbf{Host-language expressions are oracular.} Arithmetic, field projections,
and comparisons are evaluated by the embedded JavaScript engine (Boa); the \pact
type checker does not model the JS type lattice.  \item \textbf{No parametric
polymorphism.} Workflow signatures are monomorphic.  \end{itemize}

\section{Baseline Comparison Details}
\label{app:baselines}

Qualitative comparison between \pact and the three baselines on the clinical-trial workflow.

\begin{description} \item[Language-level type discipline.]
\begin{sloppypar}\pact enforces structural types on \texttt{gen} outputs at
declaration sites and rechecks after materialization. LangGraph carries no type
information between nodes beyond programmer-authored Python annotations. LMQL
constrains the \emph{shape} of a single LLM output at decoding time but does not
track the type across calls. DSPy's signature syntax is the closest in spirit
but operates at the module level, not the workflow level.\end{sloppypar}
\item[Per-invocation authorization.] \begin{sloppypar}\pact's \texttt{enforce} +
\texttt{as}-clause pattern statically requires the principal to be declared in
$\mathrm{dom}(\MPol)$ at every \texttt{gen}/\allowbreak\texttt{agent} site, and
routes the resulting file/directory operations through a runtime Cedar monitor
with $\mathsf{Read}/\mathsf{Write}/\mathsf{Exec}$ actions. In the three
baselines, authorization is either at the process boundary or authored by hand
in the node body; no baseline supports a forbid-only, root-safe install
discipline that monotonically shrinks the allow-set.\end{sloppypar}
\item[Self-healing discipline.] \pact's retry is driven by the violated type
constraint (type shape mismatch) which is written back into the prompt. DSPy
includes a feedback mechanism through its optimizer, but it operates at training
time rather than per-invocation. LangGraph and LMQL have no built-in typed
retry.  \item[Formal guarantees.] \pact has a mechanized per-step soundness
(\sect{sec:sem-correctness}). The baselines have no formal model.
\end{description}

\section{Full Reduction Rules} \label{app:rules}

This appendix lists the reduction rules elided from \sect{sec:sem-reduction} --- the remaining pure-core root rules, the ask/say rules, closure application, and the structural congruence rules --- followed by the proofs of Theorems~\ref{thm:pact-soundness}, \ref{thm:pact-progress}, and \ref{thm:pact-term}. Left-to-right call-by-value evaluation order is enforced throughout by the structural congruence rules. The substitution $\Subst{e}{v}$ denotes the capture-avoiding substitution of $v$ for $x$ in $e$; for a closure with parameters $\overline{x_i}$, simultaneous substitution $e[\overline{v_i/x_i}]$ is used.

\subsection{Let-Bind, Enforce-Install, and the Oracle Rules}
\label{app:rules-oracle}

Materialization at healable types --- the primitives $\Tschema$,
$\Tpolicy$, $\Tarrow{T}{T'}$, and any record or array containing a
healable component --- happens \emph{only} inside a
$\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{p}$ redex; every transition involving
$\Gen{\tau}{s}{\MPrin}$ fires under such a wrapping let. Two auxiliary
judgments classify the rules below. The predicate $\Healable{\tau}$
selects self-healing opportunities:
\begin{align*}
\Healable{\Tschema} &= \mathsf{true} \\
\Healable{\Tpolicy} &= \mathsf{true} \\
\Healable{\Tarrow{T_1}{T_2}} &= \mathsf{true} \\
\Healable{\Trec{\overline{x_i{:}T_i}}} &= \exists~i.\,\Healable{T_i} \\
\Healable{\Tarr{T}} &= \Healable{T} \\
\Healable{\Tunit} = \Healable{\Tbool} &= \mathsf{false} \\
\Healable{\Tnum} = \Healable{\Tstring} &= \mathsf{false}
\end{align*}
A record or array containing a healable component is itself healable, since the nested materialization site participates in self-healing. The runtime type-check is the single-valued judgment $\Gamma \vdash t : \tau \Rightarrow r$ with $r \in \{\mathsf{ok}, \mathsf{err}(\epsilon)\}$: for closed values $t = v$ the context is $\emptyset$ (runtime re-check of an oracle response), and for continuations $t = p$ it is $\Gext{x}{\tau}$ (the check that drives self-heal). $\TJerr{\Gext{x}{\tau}}{p}{\tau'}{\epsilon}$ returns the typing error string $\epsilon$ that self-healing feeds back to the oracle. We write $\StType{v}{\tau}$ for the ok-branch on closed values (used in rules where $\epsilon$ is not consumed). The static authorization premise $\PrinDeclared{\MPol}{\MPrin}$ holds when the principal $\MPrin$ named at the redex is declared in $\MPol$; success rules carry it as a premise, $\rulename{Gen-PolHeal}$ and $\rulename{Agent-PolHeal}$ fire on its negation, and a redex with no explicit \texttt{as} clause has $\MPrin = \texttt{root}$ inserted by elaboration.

\begin{sloppypar}
The rules split into three groups. (i) The non-healable group ---
$\rulename{Let-Bind}$, $\rulename{Let-Cong}$, $\rulename{Gen-Ok}$, and
$\rulename{Gen-TypeErr}$ --- handles base-typed materialization without
retry: success flushes $\MErr$ to $\emptyset$ and substitutes the
checked $v$; an ill-typed response steps to $\errV$, the fail-closed
sink. (ii) The all-$\tau$ rule $\rulename{Gen-PolHeal}$ retries on
principal denial ($\lnot\PolAllow{\MPol}{\MPrin}{\texttt{gen}}$) within
budget, appending the Cedar-supplied $\epsilon$. (iii) The
continuation-driven self-heal pairs --- the parametric pair
$\rulename{Healable-Ok}$, $\rulename{Healable-Heal}$ at any
$\Healable{\tau}$, with named Schema and Arrow specializations
($\rulename{Schema-Ok}$, $\rulename{Schema-Heal}$,
$\rulename{Arrow-Ok}$, $\rulename{Arrow-Heal}$) and the same
parametric form covering the Policy case and nested compound healable
types such as $\Tarr{\Tschema}$ from the clinical-trial
\texttt{Array[crf]} pattern --- together with $\rulename{Gen-Budget}$
form the type-driven self-heal pattern. All such rules require an
authorized principal and a well-typed $v$; they diverge on whether the
\emph{continuation} $p$ type-checks under $\Gext{x}{\tau}$. On success
the heal context is flushed and $\Subst{p}{v}$ commits via
capture-avoiding substitution (for $\Tarrow{\cdots}{\cdot}$, this
substitutes a workflow value into the continuation). On continuation failure the returned
$\epsilon$ is appended so the next oracle call reads the precise
continuation mismatch produced by the check; budget exhaustion at a
healable $\tau$ steps to $\errV$ via $\rulename{Gen-Budget}$. Under
truthful $\mathcal{O}$, T4c shows that $\rulename{Healable-Ok}$ or
$\rulename{Healable-Heal}$ fires within budget; the runtime never
observes a stall at a healable $\tau$ given an authorized principal.
The Policy case is handled by the parametric pair, since
$\Healable{\Tpolicy} = \mathsf{true}$, and follows as a corollary of the parametric healable-progress lemma.
\end{sloppypar}

\begin{sloppypar}
The agent rules ($\rulename{Agent-Success}$, $\rulename{Agent-PolHeal}$)
remain standalone: $\Agent{s}{\MPrin}$ materializes $\Tstring$, which
is non-healable, so it follows the non-healable path of returning a
checked string on success and retrying on principal denial. Each rule
names its operator directly in the conclusion, so no separate action
tag appears in the reduction state. The budget is measured by
$|\MErr|$: every entry is an $\ErrEv{\cdot}$, and a successful oracle
step flushes $\MErr$ to $\emptyset$, so the count resets to zero
naturally. The premise $\InstallOk{p}$ holds when every rule of $p$ has
effect \texttt{forbid} and no rule targets \texttt{root};
$\Install{\MPol}{p}$ then merges $p$ with $\MPol$ and the
policy-install monotonicity lemma guarantees
$\Allow{\MPol'} \subseteq \Allow{\MPol}$.
\end{sloppypar}

A configuration that lacks an outgoing transition is terminal; the language has no $\mathsf{errTerm}$ constructor, so stuckness is detected by exhausting the rule set rather than by an explicit error term. The fail-closed sink $\errV$ is a value (so $\mathsf{isValue}(\errV)$ and the configuration is terminal) and carries no runtime type by design --- T2 below excludes it from the typeable-value statement.

There is one self-heal pattern per kind of materialization. The let-centric
$\rulename{Schema-Heal}$ together with the parametric $\rulename{Healable-Heal}$
(which uniformly covers Arrow and Policy targets), and the rules
\allowbreak$\rulename{Gen-PolHeal}$, \allowbreak$\rulename{Gen-TypeErr}$,
\allowbreak$\rulename{Gen-Budget}$ together cover every \texttt{gen} self-heal
path; $\Agent{\cdot}{\cdot}$ is covered by $\rulename{Agent-Success}$ and
\allowbreak$\rulename{Agent-PolHeal}$. The plain $\Enforce{\cdot}$ form has no
separate heal rule: a $\Tpolicy$ value that fails to install is delivered by an
outer healable \texttt{let}, which retries via $\rulename{Healable-Heal}$ at
$\Tpolicy$. Closure application has no oracle component, so it needs none.

\subsection{Mutable-State Rules}
\label{app:rules-mut}

The following rules are the only ones that read or write the store $\MStore$; all other rules in \sect{app:rules} leave $\MStore$ unchanged and elide it.

\begin{figure*}[t]
\small
\begin{mathpar}
\inferrule*[right=\rulename{VarDecl-Cong}]
  {\MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{var}\;x{:}\tau = e_1\;\mathtt{in}\;e_2} \step \MConfigS{\MErr'}{\MPol'}{\MStore}{\mathtt{var}\;x{:}\tau = e_1'\;\mathtt{in}\;e_2}}

\inferrule*[right=\rulename{VarDecl-Bind}]
  {\mathsf{isValue}(v) \\ \StType{v}{\tau}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{var}\;x{:}\tau = v\;\mathtt{in}\;e_2} \step \MConfigS{\MErr}{\MPol}{\MStore[x \mapsto (\tau,v)]}{e_2}}

\inferrule*[right=\rulename{Assign-Cong}]
  {\MConfig{\MErr}{\MPol}{e} \step \MConfig{\MErr'}{\MPol'}{e'}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x \mathrel{:=} e} \step \MConfigS{\MErr'}{\MPol'}{\MStore}{x \mathrel{:=} e'}}

\inferrule*[right=\rulename{Assign-Write}]
  {\MStore(x) = (\tau, v_{\mathit{old}}) \\ \mathsf{isValue}(v) \\ \StType{v}{\tau}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x \mathrel{:=} v} \step \MConfigS{\MErr}{\MPol}{\MStore[x \mapsto (\tau,v)]}{\Tunit}}

\inferrule*[right=\rulename{VarRead}]
  {\MStore(x) = (\tau, v)}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x} \step \MConfigS{\MErr}{\MPol}{\MStore}{v}}
\end{mathpar}
\caption{Mutable-state rules for declarations, assignments, and reads over the typed store $\MStore$.}
\label{fig:rules-mut}
\end{figure*}

$\rulename{VarDecl-Bind}$ and $\rulename{Assign-Write}$ carry the runtime-type side condition $\StType{v}{\tau}$, so the store invariant $\forall~x,\tau,v.\; \MStore(x) = (\tau, v) \Rightarrow \StType{v}{\tau}$ (a conjunct of $\WF{\cdot}$) is preserved by every write. $\rulename{VarRead}$ resolves a variable reference $x$ through the store and steps deterministically when $x$ is declared. Mutable variables use named keys in $\MStore$ and use a separate name-store from the capture-avoiding substitution underlying \texttt{let} and closure application, keeping the two binding mechanisms orthogonal.

\section{Proofs of Theorems~\ref{thm:pact-soundness} and \ref{thm:pact-progress}}
\label{app:proofs}
\label{app:soundness}

Each proof is by induction on the derivation of $C \step C'$, case-splitting on
the step constructor.

\paragraph{Scope.} T1--T4c are mechanized and depend only on standard
meta-axioms; a build-time guard rejects any further axiom. The trace-level
termination argument (Theorem~\ref{thm:pact-term}) is prose-only, though its
per-step ingredients are mechanized. Progress is conditional on the oracle
assumption: when $\mathcal{O}$ is eventually-truthful, T4c fires
$\rulename{Healable-Ok}$ within budget; otherwise the trace commits, sinks to
$\errV$, or pauses at an oracle redex. Static type checking is black-boxed; the
Rust type checker is not part of the mechanized artifact. The threat-model gaps
of \sect{sec:design-threat} are outside the scope of soundness.

\paragraph{Notation.} The policy-install operation $\Install{\cdot}{\cdot}$ is
\emph{partial}; the notation $\MPol' = \Install{\MPol}{p}$ is shorthand for
``on the branch where installation succeeds.''

\subsection{T1: WF-preservation}
\label{app:proof-sound-a}

The invariant is the conjunction $|\MErr| \le \MRetry$ and the store well-typedness clause $\forall~x,\tau,v.\;\MStore(x) = (\tau,v) \Rightarrow \StType{v}{\tau}$. The proof is by cases on the step constructor; the non-trivial cases are:

\begin{sloppypar}
\paragraph{Case $\rulename{Ask}$, $\rulename{Agent-Success}$,
$\rulename{Gen-Ok}$, $\rulename{Schema-Ok}$, $\rulename{Arrow-Ok}$,
$\rulename{Healable-Ok}$.} The target heal context is $\emptyset$; $|\emptyset|
= 0 \le \MRetry$ by definition.
\end{sloppypar}

\begin{sloppypar}
\paragraph{Case $\rulename{Schema-Heal}$, $\rulename{Arrow-Heal}$,
$\rulename{Healable-Heal}$, $\rulename{Gen-PolHeal}$,
$\rulename{Agent-PolHeal}$.} The rule's own premise $|\MErr \cdot
\ErrEv{\epsilon}| \le \MRetry$ is exactly the required invariant on $C'$.
\end{sloppypar}

\paragraph{Case $\rulename{Gen-TypeErr}$, $\rulename{Gen-Budget}$.} $\MErr$ is unchanged, so the invariant is inherited.

\paragraph{Pure-core and congruence cases.} $\MErr$ is either unchanged or inherits $\le \MRetry$ from the inductive hypothesis on the inner step.

\paragraph{Case $\rulename{VarDecl-Bind}$, $\rulename{Assign-Write}$.} The rule premise $\StType{v}{\tau}$ extends or rebinds $\MStore$ at $(\tau, v)$, preserving the store-typing invariant pointwise; $\MErr$ is unchanged.

\paragraph{Case $\rulename{VarRead}$.} The store invariant on the input configuration directly supplies $\StType{v}{\tau}$ for the read value; $\MErr$ and $\MStore$ are unchanged.

The Arrow and Policy success/heal cases are covered uniformly by the parametric
$\rulename{Healable-Ok}$, $\rulename{Healable-Heal}$ rules and dispatch through
the same proof case.

\subsection{T2: Staged materialization}
\label{app:proof-sound-b}

If the step $C \step C'$ produces a value $v$ at $C'$'s expression slot and $v \neq \errV$, then $\exists~\tau.\ \StType{v}{\tau}$. The proof factors through a value lemma: $\mathsf{isValue}(v) \wedge v \neq \errV \Rightarrow \exists~\tau.\ \StType{v}{\tau}$, which enumerates the value constructors.

\paragraph{Base literals and schema/policy values.} Each base literal carries a canonical runtime type (unit, bool, number, string); schema and policy values carry their respective $\Tschema$ and $\Tpolicy$ witnesses.

\paragraph{Closures.} A closure admits an arrow type of matching arity. The T2 witness instantiates all argument and return types to $\Tunit$, giving $\tau = \Tarrow{\overline{\Tunit}}{\Tunit}$ --- the precise types of the body are irrelevant to T2's existential claim.

\paragraph{Records and arrays.} Record and array constructors instantiate witnesses at an empty field list and element type $\Tunit$, respectively, again as existential witnesses.

\paragraph{$\errV$ exclusion.} The fail-closed sink $\errV$ is excluded by hypothesis. By design, $\errV$ carries no runtime type --- it is the terminal value produced by $\rulename{Gen-TypeErr}$ and $\rulename{Gen-Budget}$ to signal an irreparable failure.

\noindent T2 applies uniformly at every reduced expression that is not the sink.

\subsection{T3: Policy monotonicity}
\label{app:proof-sound-c}

\begin{sloppypar}
$\MPol$ is updated only by $\rulename{Enforce-Install}$, whose target policy $\MPol' = \Install{\MPol}{p}$ satisfies $\Allow{\MPol'} \subseteq \Allow{\MPol}$ by the policy-install monotonicity lemma. All other rules leave $\MPol$ unchanged, giving $\Allow{\MPol'} = \Allow{\MPol}$ and hence the reflexive subset.
\end{sloppypar}

\subsection{T4a: Standalone gen is a non-redex}
\label{app:proof-prog-a}

\begin{sloppypar}
A bare $\Gen{\tau}{s}{\MPrin}$ is not a transition head: every reduction rule
targets either a different syntactic form or a congruence context that does not
match a standalone $\Gen{\tau}{s}{\MPrin}$. By constructor distinctness no rule
fires, so the configuration has no successor regardless of the budget.
Materialization always occurs inside a
$\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{\cdot}$ redex, where
$\rulename{Gen-Budget}$ fires when $|\MErr| > \MRetry$ and converts the
configuration to the fail-closed sink $\errV$.
\end{sloppypar}

\subsection{T4b: Truthful success at Schema}
\label{app:proof-prog-b}

\begin{sloppypar}
Fix a let-redex $\Let{\Tschema}{\Gen{\Tschema}{s}{\MPrin}}{p_0}$ with $\MPrin
\in \mathrm{dom}(\MPol)$ and continuation
$\TJok{\Gext{x}{\Tschema}}{p_0}{\tau'}$. By \defn{def:oracle-truthful} there
exists some $\MErr'$ with $|\MErr'| \le \MRetry$ and a value $v$ with
$\Oracle{s}{\MErr'}{\Tschema} \ni v$, $\mathsf{isValue}(v)$, and
$\StType{v}{\Tschema}$. Note the witness may appear at \emph{some} heal context
$\MErr'$, not necessarily the caller's current $\MErr$. Apply
$\rulename{Healable-Ok}$ at $\tau = \Tschema$ (the $\Healable{\Tschema}$ side
condition is immediate): all premises are discharged, so
$\MConfig{\MErr'}{\MPol}{\Let{\Tschema}{\Gen{\Tschema}{s}{\MPrin}}{p_0}} \step
\MConfig{\emptyset}{\MPol}{\Subst{p_0}{v}}$.

The Arrow analogue follows the identical pattern at
$\tau = \Tarrow{T_1 \to \cdots \to T_n}{T'}$, and the Policy analogue at
$\tau = \Tpolicy$ (discharged by $\Healable{\Tpolicy} = \mathsf{true}$). All
three are corollaries of the single parametric rule $\rulename{Healable-Ok}$.
The agent variant uses $\rulename{Agent-Success}$ on standalone
$\Agent{s}{\MPrin}$. For nested healable types such as $\Tarr{\Tschema}$ (the
clinical-trial \texttt{Array[crf]} pattern), the $\Healable{\Tarr{\Tschema}}$
premise is discharged by the recursive definition $\Healable{\Tarr{T}} =
\Healable{T}$.
\end{sloppypar}

\subsection{T4c: Progress-gen-Schema, Progress-agent}
\label{app:proof-prog-c}

\begin{sloppypar}
T4c strengthens T4b by requiring the oracle witness at the caller's
\emph{current} heal context $\MErr$ rather than at an existential $\MErr'$.
Given $\MPrin \in \mathrm{dom}(\MPol)$ and
$\TJok{\Gext{x}{\Tschema}}{p_0}{\tau'}$, the conclusion $\exists~C'.\ C \step
C'$ follows by applying $\rulename{Healable-Ok}$ at $\tau = \Tschema$ (resp.\
$\rulename{Agent-Success}$ for standalone agents) exactly as in T4b but without
the intermediate ``reach some $\MErr'$'' step. Arrow and Policy analogues
follow identically via $\rulename{Healable-Ok}$.
\end{sloppypar}

\subsection{Termination}
\label{app:term}

\begin{theorem}[Termination]
\label{thm:pact-term}
Let $C_0 = \MConfig{\emptyset}{\MPol_0}{e_0}$ be a well-formed initial configuration. Assume every \texttt{while}/\texttt{for} loop in $e_0$ terminates. Then every trace from $C_0$ either reaches a value configuration in finitely many steps (with the value either an ordinary $\mathsf{isValue}$ or the fail-closed sink $\errV$), or gets stuck at some oracle redex whose oracle never produces a well-typed, authorized response within budget. There is no ill-formed intermediate configuration. If $\mathcal{O}$ is eventually-truthful at every oracle site encountered along the trace, the trace terminates in an ordinary value configuration.
\end{theorem}

The semantics defines five possible outcomes for any terminating run:
\begin{itemize}
  \item \emph{success} (reduction reaches a value configuration),
  \item \emph{type rejection} (a non-healable \texttt{gen} produced an ill-typed value and fired $\rulename{Gen-TypeErr}$, or the budget exhausted and fired $\rulename{Gen-Budget}$),
  \item \emph{policy rejection} ($\rulename{Gen-PolHeal}$ consumed the full budget),
  \item \emph{policy-install failure} (the deny-all guard or action-type check rejected an \texttt{enforce} policy after $\MRetry$ retries), or
  \item \emph{oracle stall} (every oracle response inside budget was ill-typed at a healable $\tau$, so no rule fired and the trace got stuck).
\end{itemize}

\label{app:proof-term}
The argument is on paper. Between two successive oracle firings, only pure-core and congruence rules fire. Pure-core reduction terminates on any expression whose $\texttt{while}$ and $\texttt{for}$ loops terminate (assumed of $e_0$); substitution is finitary and preserves the termination measure on loop-free code. At each oracle firing either $\MErr$ is flushed to $\emptyset$ (success, resetting $|\MErr|$ to $0$ by T1), or an $\ErrEv{\cdot}$ extends $\MErr$ by one, or one of the hard-error rules ($\rulename{Gen-TypeErr}$ at non-healable $\tau$, $\rulename{Gen-Budget}$) fires and the configuration steps to $\errV$ --- a terminal value with no outgoing transition. At a healable $\tau$ an ill-typed oracle response has no transition: the configuration is stuck, and the trace halts there without progressing to $\errV$ (this is the \emph{oracle stall} case of the error taxonomy). After at most $\MRetry + 1$ heals at a single oracle site, the configuration is either pushed into the budget-exhausted regime where $\rulename{Gen-Budget}$ fires deterministically, or has already succeeded. Under eventually-truthful $\mathcal{O}$, truthful-success is guaranteed at every site; since the number of oracle sites reached along the trace is finite (bounded by $|e_0|$ plus finitely many $\rulename{Eval-Success}$ expansions), the trace terminates in an ordinary value configuration. Otherwise the trace either terminates in an ordinary value, terminates in $\errV$, or gets stuck at an oracle redex whose oracle never produces a witness within budget; no ill-formed configuration is ever reached.
