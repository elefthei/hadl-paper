\section{Threat Model and Out-of-Scope Concerns}
\label{app:threat}

\paragraph{Trust boundaries.} \hadl treats the programmer-authored program text, the static type checker, the runtime re-checker, and the installed Cedar decision procedure as trusted. It treats every LLM output, whether a value, a schema, a workflow AST, or a policy, as untrusted. The retry budget $\MRetry$ declared by \texttt{@retries} is a bound on how many untrusted attempts the runtime will consume before failing closed on a given generation site.

\paragraph{What is caught.} Validation rejects an LLM output that fails the structural checks of the annotated type: a schema whose $\sigma$-image is not representable, a workflow AST that fails to type-check at its use site, a policy that triggers the deny-all guard or references an unknown action type. Every \texttt{gen} and \texttt{agent} invocation is checked against the installed policy before the LLM is called, under the explicit principal named by the surrounding \texttt{as} clause; a denial at this check becomes an $\ErrEv{\cdot}$ event and triggers retry. Append-only merging guarantees that no \texttt{enforce} step can relax an already-installed restriction.

\paragraph{What is not caught.} Cedar authorizes at the granularity of a \texttt{(principal, action, resource)} triplet where the action is drawn from the fixed set $\{\texttt{gen},\texttt{agent}\}$ and the resource is the call site. It does not model the downstream side effects of an authorized \texttt{agent} call, the contents of its prompt, or the data returned by the LLM. Four categories of threat are therefore outside the current enforcement model. First, \emph{prompt injection} against the LLM itself: a \texttt{gen} call whose prompt is composed with untrusted \texttt{ask} input can still be steered by an adversary who controls the user. Second, \emph{semantic misalignment} of generated policies: the deny-all guard and action-type check filter out policies that are syntactically pathological, but an LLM-generated policy that is merely \emph{over-permissive} with respect to operator intent will install and appear well-formed. Third, \emph{data exfiltration} via \texttt{say}, where an already-authorized workflow emits sensitive values to the human. Fourth, \emph{side-effecting js blocks}, which execute arbitrary JavaScript under the host runtime's ambient authority. These gaps are out of scope for gradual validation as we define it; the append-only discipline prevents policy relaxation but cannot recover from a malicious or inappropriate \emph{initial} policy.

\section{Type-System Limitations}
\label{app:limitations}

The type system deliberately makes several simplifications to keep gradual validation tractable.

\begin{itemize}
  \item \textbf{Arrays are homogeneously element-typed.} \texttt{array[T]} requires every element to have type $T$; there are no tuple types and no heterogeneous arrays.
  \item \textbf{Records check declared fields only.} A record type $\{\overline{x_i{:}T_i}\}$ requires every listed field to be present with the listed type, but the runtime value may carry additional fields. There is no row polymorphism, no width subtyping beyond the \texttt{Schema}-as-top rule, and no structural subtyping between record types.
  \item \textbf{No unions or intersections.} There are no sum types, union types, or refinement types. Values whose shape depends on runtime choice must flow through \texttt{Schema} and be re-checked at use.
  \item \textbf{\texttt{Schema} is top during static checking.} Both $T \le \texttt{Schema}$ and $\texttt{Schema} \le T$ hold for every $T$, so a \texttt{Schema}-typed value passes every static field-access and \texttt{eval} check; field-level validity is deferred to the post-materialization re-checker and, for compound schemas that $\sigma$ maps back to \texttt{Schema}, to the JavaScript engine at runtime.
  \item \textbf{\texttt{js} blocks are oracular.} Inline \texttt{js} expressions produce a value whose runtime type is whatever \texttt{RtType} accepts; the \hadl type checker does not model the JS type lattice.
  \item \textbf{No parametric polymorphism.} Workflow signatures are monomorphic.
\end{itemize}

\section{Evaluation Methodology}
\label{app:eval-method}

\paragraph{Workflow corpus.} The six workflows are the clinical-trial task of \figr{fig:clinical-trial}, the car scorer of \sect{sec:overview}, and four additional tasks: a customer-support triage workflow (\texttt{Schema}, \texttt{Policy}), a code-review agent (arrow type, \texttt{Policy}), a financial-compliance report (\texttt{Schema}, \texttt{Policy}), and a research-paper summarizer (\texttt{Schema}).

\paragraph{Models.} Each workflow is run against three LLMs: GPT-4.1, Claude-Sonnet-4, and the default model exposed by the GitHub Copilot SDK at the time of evaluation. All three are accessed through the same \hadl oracle interface.

\paragraph{Runs.} Each (workflow, model) pair is executed $K$ times with distinct seeds and distinct synthetic user inputs drawn from a per-workflow input template, for a total of $6 \times 3 \times K$ runs. Workflows that depend on human input (\texttt{ask}) are driven by a scripted stub.

\paragraph{Retry budget.} All experiments fix $\MRetry = 3$, matching the \texttt{@retries 3} pragma on Figure~\ref{fig:clinical-trial}.

\paragraph{Error taxonomy.} Every terminating run is classified as one of \emph{success} (reduction reaches a value configuration); \emph{type rejection} (a $\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{\cdot}$ at non-healable $\tau$ produced an ill-typed value and fired $\rulename{Let-Gen-TypeError}$, or budget exhausted at any $\tau$ fired $\rulename{Let-Gen-BudgetError}$); \emph{policy rejection} ($\rulename{Let-Gen-HealPol}$ consumed the full budget); \emph{policy-install failure} (deny-all guard or action-type check rejected an \texttt{enforce} policy after $\MRetry$ retries); \emph{oracle stall} (at a healable $\tau$, every oracle response inside budget was ill-typed at $\tau$, so no rule fired and the trace got stuck --- distinct from termination at $\errV$); or \emph{runtime error} (non-oracle exception: JS engine failure, network error). A \emph{type rejection} run terminates by reaching a configuration whose expression slot is the fail-closed sink $\errV$.

\paragraph{Determinism controls.} For each (workflow, model) pair we record the full prompt sequence and the full oracle response trace. This lets us replay a failure deterministically for the post-mortem and makes the token-cost numbers reproducible.

\section{Baseline Comparison Details}
\label{app:baselines}

Qualitative comparison between \hadl and the three baselines on the clinical-trial workflow.

\begin{description}
  \item[Language-level type discipline.] \hadl enforces structural types on \texttt{gen} outputs at declaration sites and rechecks after materialization. LangGraph carries no type information between nodes beyond programmer-authored Python annotations. LMQL constrains the \emph{shape} of a single LLM output at decoding time but does not track the type across calls. DSPy's signature syntax is the closest in spirit but operates at the module level, not the workflow level.
  \item[Per-invocation authorization.] \hadl's \texttt{enforce} + \texttt{as}-clause pattern produces a Cedar check at every \texttt{gen}/\texttt{agent} call site. In the three baselines, authorization is either at the process boundary or authored by hand in the node body; no baseline supports append-only policy merging.
  \item[Self-healing discipline.] \hadl's retry is driven by the violated constraint itself (type shape, Cedar denial) which is written back into the prompt. DSPy includes a feedback mechanism through its optimizer, but it operates at training time rather than per-invocation. LangGraph and LMQL have no built-in typed retry.
  \item[Formal guarantees.] \hadl has a mechanized per-step soundness (\sect{sec:sem-correctness}). The baselines have no formal model.
\end{description}

\section{Full Reduction Rules}
\label{app:rules}

This appendix lists the reduction rules elided from \sect{sec:sem-reduction} --- the remaining pure-core root rules, the ask/say rules, closure application, and the structural congruence rules --- followed by the proofs of Theorems~\ref{thm:hadl-soundness}, \ref{thm:hadl-progress}, and \ref{thm:hadl-term}. Left-to-right call-by-value evaluation order is enforced throughout by the structural congruence rules. The substitution $\Inst{e}{v}$ denotes the capture-avoiding instantiation of de-Bruijn index $0$ in $e$ with the value $v$ (decrementing all other free indices); for a closure body with $n$ binders, simultaneous substitution $e[v_0/0,\dots,v_{n-1}/(n{-}1)]$ is used.

\subsection{Let-Bind, Enforce-Install, and the Oracle Rules}
\label{app:rules-oracle}

Materialization at the three healable types --- $\Tschema$, $\Tpolicy$, and $\Tarrow{T}{T'}$ --- happens \emph{only} inside a $\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{p}$ redex. Standalone $\Gen{\tau}{s}{\MPrin}$ has no transition. Two auxiliary judgments classify the rules below. The predicate $\Healable{\tau}$ selects materialization targets:
\begin{align*}
\Healable{\Tschema} &= \mathsf{true} & \Healable{\Tpolicy} &= \mathsf{true} & \Healable{\Tarrow{T_1}{T_2}} &= \mathsf{true} \\
\Healable{\Trec{\overline{x_i{:}T_i}}} &= \exists i.\,\Healable{T_i} & \Healable{\Tarr{T}} &= \Healable{T} & \Healable{\Tunit} = \Healable{\Tbool} = \Healable{\Tnum} = \Healable{\Tstring} &= \mathsf{false}
\end{align*}
A record or array containing a healable component is itself healable, since the nested materialization site still requires self-healing. The runtime type-check is the single-valued judgment $\Gctx \vdash t : \tau \Rightarrow r$ with $r \in \{\mathsf{ok}, \mathsf{err}(\epsilon)\}$: it applies to values $t = v$ (runtime re-check of an oracle response) \emph{and} to expressions $t = p$ (continuation check that drives self-heal, per \texttt{hadl-formal.md}). $\TJerr{\Gctx}{t}{\tau}{\epsilon}$ returns the typing error string $\epsilon$ that self-healing feeds back to the oracle. The context $\Gctx$ binds type-alias names introduced by \texttt{type} declarations; self-heal rules extend it to $\Gext{x}{\tau}$ when type-checking the continuation. We write $\RtType{v}{\tau}$ for the ok-branch $\TJok{\emptyset}{v}{\tau}$ on closed values (used in rules where $\epsilon$ is not consumed).

\begin{mathpar}
\inferrule*[right=\rulename{Let-Bind}]
  {\mathsf{isValue}(v) \\ \TJok{\Gctx}{v}{\tau}}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{v}{e}} \step \MConfig{\MErr}{\MPol}{\Inst{e}{v}}}

\inferrule*[right=\rulename{Let-Cong-Nonheal}]
  {\lnot \Healable{\tau} \\ \MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{e_1}{e_2}} \step \MConfig{\MErr'}{\MPol'}{\Let{\tau}{e_1'}{e_2}}}

\inferrule*[right=\rulename{Let-Gen-Success-Nonheal}]
  {\lnot \Healable{\tau} \\
   \PolOk{\MPol}{\MPrin}{\texttt{gen}} \\

   \TJok{\Gctx}{v}{\tau}}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{e}} \step \MConfig{\emptyset}{\MPol}{\Inst{e}{v}}}

\inferrule*[right=\rulename{Let-Gen-TypeError}]
  {\lnot \Healable{\tau} \\

  {\MConfig{\MErr}{\MPol}{\Let{\Tschema}{\Gen{\Tschema}{s}{\MPrin}}{p}} \step \MConfig{\emptyset}{\MPol}{\Inst{p}{v}}}

\inferrule*[right=\rulename{Let-Gen-Heal-Schema}]
  {\PolOk{\MPol}{\MPrin}{\texttt{gen}} \\
   \Oracle{s}{\MErr}{\Tschema} \ni v \\
   \TJok{\Gctx}{v}{\Tschema} \\

   \TJok{\Gext{x}{\Tarrow{T_1,\ldots,T_n}{T'}}}{p}{\tau''}}
  {\MConfig{\MErr}{\MPol}{\Let{\Tarrow{T_1,\ldots,T_n}{T'}}{\Gen{\Tarrow{T_1,\ldots,T_n}{T'}}{s}{\MPrin}}{p}} \step \MConfig{\emptyset}{\MPol}{\Inst{p}{v}}}

\inferrule*[right=\rulename{Let-Gen-Heal-Arrow}]

  {\MConfig{\MErr}{\MPol}{\Let{\Tarrow{T_1,\ldots,T_n}{T'}}{\Gen{\Tarrow{T_1,\ldots,T_n}{T'}}{s}{\MPrin}}{p}} \step \MConfig{\MErr \cdot \ErrEv{\epsilon}}{\MPol}{\Let{\Tarrow{T_1,\ldots,T_n}{T'}}{\Gen{\Tarrow{T_1,\ldots,T_n}{T'}}{s}{\MPrin}}{p}}}

\inferrule*[right=\rulename{Let-Gen-Success-Healable}]

   \Oracle{s}{\MErr}{\tau} \ni v \\ \TJok{\Gctx}{v}{\tau} \\
   \TJok{\Gext{x}{\tau}}{p}{\tau''}}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{p}} \step \MConfig{\emptyset}{\MPol}{\Inst{p}{v}}}

\inferrule*[right=\rulename{Let-Gen-Heal-Healable}]
  {\Healable{\tau} \\ \PolOk{\MPol}{\MPrin}{\texttt{gen}} \\
   \Oracle{s}{\MErr}{\tau} \ni v \\ \TJok{\Gctx}{v}{\tau} \\
   \TJerr{\Gext{x}{\tau}}{p}{\tau''}{\epsilon} \\
   \retries{\MErr \cdot \ErrEv{\epsilon}} \le \MRetry}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{p}} \step \MConfig{\MErr \cdot \ErrEv{\epsilon}}{\MPol}{\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{p}}}

\inferrule*[right=\rulename{Enforce-Install}]
  {\Install{\MPol}{p} = \MPol'}
  {\MConfig{\MErr}{\MPol}{\Enforce{p}} \step \MConfig{\MErr}{\MPol'}{\Tunit}}

\inferrule*[right=\rulename{Agent-Success}]
  {\PolOk{\MPol}{\MPrin}{\texttt{agent}} \\
   \Oracle{s}{\MErr}{\Tstring} \ni v \\
   \TJok{\Gctx}{v}{\Tstring}}
  {\MConfig{\MErr}{\MPol}{\Agent{s}{\MPrin}} \step \MConfig{\emptyset}{\MPol}{v}}

\inferrule*[right=\rulename{Agent-Heal-Pol}]
  {\lnot\,\PolOk{\MPol}{\MPrin}{\texttt{agent}} \\
   \epsilon = \Explain{\MPrin}{\MPol} \\
   \retries{\MErr \cdot \ErrEv{\epsilon}} \le \MRetry}
  {\MConfig{\MErr}{\MPol}{\Agent{s}{\MPrin}} \step \MConfig{\MErr \cdot \ErrEv{\epsilon}}{\MPol}{\Agent{s}{\MPrin}}}
\end{mathpar}

The rules split into three groups. (i) The non-healable group --- $\rulename{Let-Bind}$, $\rulename{Let-Cong-Nonheal}$, $\rulename{Let-Gen-Success-Nonheal}$, and $\rulename{Let-Gen-TypeError}$ --- handles base-typed materialization with no retry: success flushes $\MErr$ to $\emptyset$, type failure steps to $\errV$ (fail-closed; there is no retry to feed the returned $\epsilon$ into). (ii) The uniform-error group --- $\rulename{Let-Gen-BudgetError}$, $\rulename{Let-Gen-HealPol}$ --- applies at \emph{any} $\tau$: budget exhaustion sinks to $\errV$, authorization denial retries within budget with a Cedar-supplied $\epsilon$. (iii) The continuation-driven self-heal pairs --- the parametric pair $\rulename{Let-Gen-Success-Healable}$/$\rulename{Let-Gen-Heal-Healable}$ at any $\Healable{\tau}$ (which subsumes the per-shape pairs $\rulename{Let-Gen-Success-Schema}$/$\rulename{Let-Gen-Heal-Schema}$ and $\rulename{Let-Gen-Success-Arrow}$/$\rulename{Let-Gen-Heal-Arrow}$ and additionally covers nested compound healable types such as $\Tarr{\Tschema}$ from the clinical-trial \texttt{Array[crf]} pattern), plus the shared $\rulename{Let-Gen-BudgetError}$ for budget exhaustion --- are the type-driven self-heal pattern. All such rules require a well-typed $v$; they diverge on whether the \emph{continuation} $p$ type-checks under $\Gext{x}{\tau}$. On success the heal context is flushed and $\Inst{p}{v}$ commits via the capture-avoiding substitution that $\rulename{Let-Bind}$ and closure application already use --- for $\Tarrow{\cdots}{\cdot}$ this is higher-order substitution of a workflow value into the continuation, mechanized in \texttt{mechanization/HADL/Syntax.lean} via the \texttt{lean-subst} framework. On continuation failure the returned $\epsilon$ is appended so the next oracle call reads the precise continuation mismatch produced by the check. If the oracle returns an ill-typed $v$ at a healable $\tau$, no rule fires --- the configuration is stuck by omission, and progress (T4c) discharges this case under eventually-truthful $\mathcal{O}$. The Policy pair is subsumed by the parametric pair: $\Healable{\Tpolicy} = \mathsf{true}$, and the segment-authorization check on $\texttt{gen}$ is exactly the existing $\PolOk{\MPol}{\MPrin}{\texttt{gen}}$ premise of $\rulename{Let-Gen-Success-Healable}$, so no separate Policy rule is required. Mechanized as $\mathtt{T4\_progress\_gen\_policy}$ in \texttt{Safety.lean}.

The agent rules ($\rulename{Agent-Success}$, $\rulename{Agent-Heal-Pol}$) remain standalone: $\Agent{s}{\MPrin}$ materializes $\Tstring$, which is non-healable, so there is no agent type-heal rule. The action tag $a$ in earlier drafts is no longer used --- each rule names its operator in the conclusion. The budget metric $\retries{\MErr}$ is simply $|\MErr|$: every entry is an $\ErrEv{\cdot}$, and a successful oracle step flushes $\MErr$ to $\emptyset$, so the metric resets to zero naturally. $\Install{\MPol}{p}$ bundles the deny-all guard, action-type validation, and append-only merge; $\mathtt{policyInstall\_shrinks}$ guarantees $\Allow{\MPol'} \subseteq \Allow{\MPol}$.

\inferrule*[right=\rulename{Eval-Fun-Cong}]
  {\MConfig{\MErr}{\MPol}{f} \step \MConfig{\MErr'}{\MPol'}{f'}}
  {\MConfig{\MErr}{\MPol}{\Eval{f}{\overline{v_i}}} \step \MConfig{\MErr'}{\MPol'}{\Eval{f'}{\overline{v_i}}}}
\end{mathpar}

None of the congruence rules carries a $\neq \mathsf{err}$ side condition: the language has no $\mathsf{errTerm}$ constructor, so a stuck configuration has no outgoing transition and its stuckness simply propagates up by absence.

There is no separate heal rule for $\Enforce{\cdot}$ or for closure application: $\Enforce{\cdot}$ whose policy value fails to install simply gets stuck (no rewind-to-gen fallback); closure application has no oracle component. The let-centric $\rulename{Let-Gen-Heal-Schema}$ together with the parametric $\rulename{Let-Gen-Heal-Healable}$ (which uniformly covers Arrow and Policy targets via $\Healable{\Tarrow{\cdots}{\cdot}}$ and $\Healable{\Tpolicy}$), and the uniform $\rulename{Let-Gen-HealPol}$/$\rulename{Let-Gen-TypeError}$/$\rulename{Let-Gen-BudgetError}$ together cover every \texttt{gen} self-heal path; $\Agent{\cdot}{\cdot}$ is covered by $\rulename{Agent-Success}$ and $\rulename{Agent-Heal-Pol}$. The fail-closed sink $\errV$ replaces the earlier $\mathsf{errTerm}$ constructor; it is a value (so $\mathsf{isValue}(\errV)$ holds and the configuration is terminal) but carries no runtime type by design (see T2 below).

\subsection{Mutable-State Rules}
\label{app:rules-mut}

The following rules are the only ones that read or write the store $\MStore$; all other rules in \sect{app:rules} leave $\MStore$ unchanged and elide it.

\inferrule*[right=\rulename{VarDecl-Cong}]

\inferrule*[right=\rulename{VarDecl-Bind}]
  {\mathsf{isValue}(v) \\ \RtType{v}{\tau}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{var}\;x{:}\tau = v\;\mathtt{in}\;e_2} \step \MConfigS{\MErr}{\MPol}{\MStore[x \mapsto (\tau,v)]}{e_2}}

\inferrule*[right=\rulename{Assign-Cong}]
  {\MConfig{\MErr}{\MPol}{e} \step \MConfig{\MErr'}{\MPol'}{e'}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x \mathrel{:=} e} \step \MConfigS{\MErr'}{\MPol'}{\MStore}{x \mathrel{:=} e'}}

\inferrule*[right=\rulename{Assign-Write}]
  {\MStore(x) = (\tau, v_{\mathit{old}}) \\ \mathsf{isValue}(v) \\ \RtType{v}{\tau}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x \mathrel{:=} v} \step \MConfigS{\MErr}{\MPol}{\MStore[x \mapsto (\tau,v)]}{\Tunit}}

\inferrule*[right=\rulename{VarRead}]
  {\MStore(x) = (\tau, v)}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{read}\;x} \step \MConfigS{\MErr}{\MPol}{\MStore}{v}}
\end{mathpar}

$\rulename{VarDecl-Bind}$ and $\rulename{Assign-Write}$ carry the runtime-type side condition $\RtType{v}{\tau}$, so the store invariant $\forall x,\tau,v.\; \MStore(x) = (\tau, v) \Rightarrow \RtType{v}{\tau}$ (a conjunct of $\WF{\cdot}$) is preserved by every write. $\rulename{VarRead}$ resolves a read through the store and cannot be stuck on a well-formed configuration where the $x$ was declared. Mutable variables are \emph{not} de-Bruijn indices: they use named keys in $\MStore$ and never interact with the capture-avoiding substitution underlying \texttt{let} and closure application.

\section{Proofs of Theorems~\ref{thm:hadl-soundness} and \ref{thm:hadl-progress}}
\label{app:proofs}

Each proof is an induction on the derivation of $C \step C'$, mirroring the Lean case split on the constructor of $\mathtt{Step}$. The mechanization is free of user-introduced axioms: $\mathtt{policyInstall\_shrinks}$ is proved by iterated application of Cedar-Lean's $\mathtt{unchanged\_deny\_when\_add\_forbid}$ theorem, and the JavaScript host evaluator is modelled by an opaque $\mathtt{jsEval} : \mathtt{JsExpr} \to \mathsf{Option}\;\mathtt{Value}$ whose stub implementation returns $\mathsf{none}$; consequently the Lean $\mathtt{jsEval\_wellTyped}$ lemma is vacuous, and the paper-level ``oracular \texttt{js}'' story is not yet axiomatized --- a faithful axiomatization would replace the stub with a single $\mathtt{js\_oracular}$ axiom and add it to the allowlist checked by $\mathtt{AxiomCheck.lean}$. The policy-install operation $\Install{\cdot}{\cdot}$ is \emph{partial} (Lean's $\mathtt{policyInstall}$ returns $\mathsf{Option}$); the paper's $\MPol' = \Install{\MPol}{p}$ notation is shorthand for ``on the branch where installation succeeds''. All theorems depend only on Lean's standard meta-axioms ($\mathtt{propext}$, $\mathtt{Classical.choice}$, $\mathtt{Quot.sound}$); no other steps are admitted.

\subsection{T1: WF-preservation}
\label{app:proof-sound-a}

The invariant is the single clause $\retries{\MErr} \le \MRetry$. The proof is by cases on the $\mathtt{Step}$ constructor in \texttt{Soundness.lean}; the non-trivial cases are:

\paragraph{Case $\rulename{Ask}$, $\rulename{Agent-Success}$, $\rulename{Let-Gen-Success-Nonheal}$, $\rulename{Let-Gen-Success-Schema}$, $\rulename{Let-Gen-Success-Arrow}$, $\rulename{Let-Gen-Success-Healable}$.} The target heal context is $\emptyset$; $\retries{\emptyset} = 0 \le \MRetry$ by definition.

\paragraph{Case $\rulename{Let-Gen-Heal-Schema}$, $\rulename{Let-Gen-Heal-Arrow}$, $\rulename{Let-Gen-Heal-Healable}$, $\rulename{Let-Gen-HealPol}$, $\rulename{Agent-Heal-Pol}$.} The rule's own premise $\retries{\MErr \cdot \ErrEv{\epsilon}} \le \MRetry$ is exactly the required invariant on $C'$.

\paragraph{Case $\rulename{Let-Gen-TypeError}$, $\rulename{Let-Gen-BudgetError}$.} $\MErr$ is unchanged, so the invariant is inherited.

\paragraph{Pure-core and congruence cases.} $\MErr$ is either unchanged or inherits $\le \MRetry$ from the inductive hypothesis on the inner step.

Because $\WF{C}$ no longer involves environment well-typedness or residual static typeability, the per-rule cases are uniformly trivial --- this is the mechanized content of $\mathtt{T1\_WF\_preservation}$. The Arrow and Policy success/heal cases are covered uniformly by the parametric $\rulename{Let-Gen-Success-Healable}$/$\rulename{Let-Gen-Heal-Healable}$ rules and dispatch through the same proof case in $\mathtt{T1\_WF\_preservation}$.

\subsection{T2: Staged materialization}
\label{app:proof-sound-b}

If the step $C \step C'$ produces a value $v$ at $C'$'s expression slot \emph{and} $v \neq \errV$, then $\exists \tau.\ \RtType{v}{\tau}$. The proof ($\mathtt{T2\_staged\_materialization}$) factors through a value-lemma ``$\mathsf{isValue}(v) \wedge v \neq \errV \Rightarrow \exists \tau.\ \RtType{v}{\tau}$'' that enumerates the value constructors. For base literals and schema/policy values the corresponding $\mathtt{vUnit}/\mathtt{vBool}/\mathtt{vNum}/\mathtt{vStr}/\mathtt{vSchema}/\mathtt{vPol}$ case applies. For closures, the $\mathtt{vClos}$ rule admits an arrow of matching arity; the T2 witness instantiates arguments and return to $\Tunit$, giving $\tau = \Tarrow{\overline{\Tunit}}{\Tunit}$ --- the precise argument and return types of the body are irrelevant to T2's existential claim. For record and array values, $\mathtt{vRec}$ and $\mathtt{vArr}$ instantiate witnesses at records with an empty field list and arrays of $\Tunit$, again as existential witnesses. The $\errV$ case is excluded by hypothesis: $\errV$ is the fail-closed sink produced by $\rulename{Let-Gen-TypeError}$ and $\rulename{Let-Gen-BudgetError}$ and intentionally has no runtime type, so a configuration whose expression slot is $\errV$ is a terminal failure state, not a typeable value. T2 is the per-step staged materialization claim and applies uniformly at every reduced expression that is not the sink.

\subsection{T3: Policy monotonicity}
\label{app:proof-sound-c}

$\MPol$ is updated only by $\rulename{Enforce-Install}$, whose target policy $\MPol' = \Install{\MPol}{p}$ satisfies $\Allow{\MPol'} \subseteq \Allow{\MPol}$ by $\mathtt{policyInstall\_shrinks}$. All other rules leave $\MPol$ unchanged and give $\Allow{\MPol'} = \Allow{\MPol}$, hence the reflexive subset. The pointwise authorization claim is immediate because $\rulename{Let-Gen-Success-Schema}$, $\rulename{Let-Gen-Success-Arrow}$, $\rulename{Let-Gen-Success-Nonheal}$, $\rulename{Let-Gen-TypeError}$, and $\rulename{Agent-Success}$ all carry $\PolOk{\MPol}{\MPrin}{\cdot}$ as a side condition (and $\rulename{Let-Gen-HealPol}$/$\rulename{Agent-Heal-Pol}$ carry its negation, which is why they exist).

\subsection{T4a: No transition for budget-exhausted standalone gen}
\label{app:proof-prog-a}

T4a is now \emph{vacuously true}: $\Gen{\tau}{s}{\MPrin}$ is not a redex. We case-split on the $\mathtt{Step}$ constructor producing any putative $C \step C'$ from $C = \MConfig{\MErr}{\MPol}{\Gen{\tau}{s}{\MPrin}}$. Every constructor either targets a different syntactic head ($\mathtt{enforceInstall}$, $\mathtt{evalSuccess}$, $\mathtt{letBind}$, $\mathtt{letCongNonheal}$, $\mathtt{letGenSuccessNonheal}$, $\mathtt{letGenSuccessSchema}$, $\mathtt{letGenSuccessArrow}$, $\mathtt{letGenTypeError}$, $\mathtt{letGenBudgetError}$, $\mathtt{letGenHealPol}$, $\mathtt{letGenHealSchema}$, $\mathtt{letGenHealArrow}$, $\mathtt{agentSuccess}$, $\mathtt{agentHealPol}$, $\mathtt{ask}$, $\ldots$), or is a congruence rule whose target structure does not match $\Gen{\tau}{s}{\MPrin}$. By constructor distinctness no rule fires, so $C$ has no successor, regardless of the budget. This is mechanized as $\mathtt{T4\_budget\_no\_heal}$ in \texttt{Safety.lean}; the original ``budget $\Rightarrow$ no heal'' content is now subsumed by $\rulename{Let-Gen-BudgetError}$, which fires \emph{instead of} the heal rules at the surrounding $\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{\cdot}$ when $\retries{\MErr} > \MRetry$ and converts the configuration to the fail-closed sink $\errV$.

\subsection{T4b: Truthful success at Schema}
\label{app:proof-prog-b}

Fix a let-redex $\Let{\Tschema}{\Gen{\Tschema}{s}{\MPrin}}{p_0}$ with $\PolOk{\MPol}{\MPrin}{\texttt{gen}}$ and a continuation $p_0$ that is statically well-typed at $\tau'$ under $\Gext{x}{\Tschema}$ (i.e.\ $\TJok{\Gext{x}{\Tschema}}{p_0}{\tau'}$). By \defn{def:oracle-truthful} there exists some $\MErr'$ with $\retries{\MErr'} \le \MRetry$ and a value $v$ with $\Oracle{s}{\MErr'}{\Tschema} \ni v$, $\mathsf{isValue}(v)$, and $\RtType{v}{\Tschema}$. Apply $\rulename{Let-Gen-Success-Healable}$ at $\tau = \Tschema$ (the $\Healable{\Tschema}$ side condition is immediate): all premises are discharged (policy, oracle, $v$:Schema$\Rightarrow$ok, $p_0$:$\tau'\Rightarrow$ok), so $\MConfig{\MErr'}{\MPol}{\Let{\Tschema}{\Gen{\Tschema}{s}{\MPrin}}{p_0}} \step \MConfig{\emptyset}{\MPol}{\Inst{p_0}{v}}$. Mechanized as $\mathtt{T4\_truthful\_success}$ in \texttt{Safety.lean}; the arrow analogue $\mathtt{T4\_truthful\_success\_arrow}$ follows the identical pattern at $\tau = \Tarrow{T_1,\ldots,T_n}{T'}$ (discharged via the universally-quantified $\mathtt{StaticTypeOK.var0}$ witness, which applies to any $\tau$). Both Schema and Arrow specializations dispatch through the single parametric rule $\rulename{Let-Gen-Success-Healable}$ in the Lean reduction relation; the agent variant is $\mathtt{T4\_truthful\_success\_agent}$ and uses $\rulename{Agent-Success}$ on standalone $\Agent{s}{\MPrin}$. The Policy analogue $\mathtt{T4\_progress\_gen\_policy}$ (\texttt{Safety.lean}) is the corresponding 1-line corollary of the parametric $\mathtt{T4\_progress\_gen\_healable}$ at $\tau = \Tpolicy$, discharged by $\Healable{\Tpolicy} = \mathsf{true}$. The parametric generalization $\mathtt{T4\_truthful\_success\_healable}$ (also in \texttt{Safety.lean}) takes any $\Healable{\tau}$ and applies $\rulename{Let-Gen-Success-Healable}$; the canonical nested case $\Tarr{\Tschema}$ (the clinical-trial \texttt{Array[crf]} pattern) is mechanized as the worked-example theorem $\mathtt{nested\_array\_of\_schema\_succeeds}$, which discharges the $\Healable{\Tarr{\Tschema}}$ premise by the recursive definition $\Healable{\Tarr{T}} = \Healable{T}$ and $\Healable{\Tschema} = \mathsf{true}$.

\subsection{T4c: Progress-gen-Schema, Progress-agent}
\label{app:proof-prog-c}

These are wrappers around T4b: they add $\PolOk{\MPol}{\MPrin}{\cdot}$ and the continuation typeability $\TJok{\Gext{x}{\Tschema}}{p_0}{\tau'}$ as explicit hypotheses and take a direct oracle witness $\exists v.\ \mathsf{isValue}(v) \wedge \RtType{v}{\Tschema} \wedge \Oracle{s}{\MErr}{\Tschema} \ni v$ at the caller's \emph{current} heal context. The conclusion $\exists C'.\ C \step C'$ follows by applying $\rulename{Let-Gen-Success-Healable}$ at $\tau = \Tschema$ (resp.\ $\rulename{Agent-Success}$, whose materialization target $\Tstring$ is non-healable so no continuation check is needed) exactly as in T4b but without the intermediate ``reach some $\MErr'$'' step. Mechanized as $\mathtt{T4\_progress\_gen}$ / $\mathtt{T4\_progress\_agent}$ in \texttt{Safety.lean}; the arrow analogue follows the same pattern at $\tau = \Tarrow{T_1,\ldots,T_n}{T'}$, again routed through $\rulename{Let-Gen-Success-Healable}$. The Policy analogue $\mathtt{T4\_progress\_gen\_policy}$ (\texttt{Safety.lean}) is the corresponding 1-line corollary of the parametric $\mathtt{T4\_progress\_gen\_healable}$ at $\tau = \Tpolicy$.

\subsection{Termination}
\label{app:term}

\begin{theorem}[Termination]
\label{thm:hadl-term}
Let $C_0 = \MConfig{\emptyset}{\MPol_0}{e_0}$ be a well-formed initial configuration. Assume every \texttt{while}/\texttt{for} loop in $e_0$ terminates. Then every trace from $C_0$ either reaches a value configuration in finitely many steps (with the value either an ordinary $\mathsf{isValue}$ or the fail-closed sink $\errV$), or gets stuck at some oracle redex whose retry budget has been exhausted without a truthful response. There is no ill-formed intermediate configuration. If $\mathcal{O}$ is eventually-truthful at every oracle site encountered along the trace, the trace terminates in an ordinary value configuration.
\end{theorem}

\label{app:proof-term}
The argument is on paper. Between two successive oracle firings, only pure-core and congruence rules fire. Pure-core reduction terminates on any expression whose $\texttt{while}$ and $\texttt{for}$ loops terminate (assumed of $e_0$); substitution is finitary and preserves the termination measure on loop-free code. At each oracle firing either $\MErr$ is flushed to $\emptyset$ (success, resetting $\retries{\cdot}$ to $0$ by T1), or an $\ErrEv{\cdot}$ extends $\MErr$ by one, or one of the hard-error rules ($\rulename{Let-Gen-TypeError}$ at non-healable $\tau$, $\rulename{Let-Gen-BudgetError}$) fires and the configuration steps to $\errV$ --- a terminal value with no outgoing transition. At a healable $\tau$ an ill-typed oracle response has no transition: the configuration is stuck, and the trace halts there without progressing to $\errV$ (this is the \emph{oracle stall} case of the error taxonomy). After at most $\MRetry + 1$ heals at a single oracle site, the configuration is either pushed into the budget-exhausted regime where $\rulename{Let-Gen-BudgetError}$ fires deterministically, or has already succeeded. Under eventually-truthful $\mathcal{O}$, truthful-success is guaranteed at every site; since the number of oracle sites reached along the trace is finite (bounded by $|e_0|$ plus finitely many $\rulename{Eval-Success}$ expansions), the trace terminates in an ordinary value configuration. Otherwise the trace either terminates in an ordinary value, terminates in $\errV$, or gets stuck at an oracle redex whose oracle never produces a witness within budget; no ill-formed configuration is ever reached.

\subsection{Auxiliary Definitions and Internal Boundaries}
\label{app:aux-defs}

\paragraph{Multi-step closure.} The reflexive-transitive closure $\step^{*}$ of $\step$ is mechanized as $\mathtt{Steps}$ (\texttt{Reduction.lean}). Composite end-to-end progress comes in two flavours: an \emph{independent} statement bundling per-redex single-step progresses ($\mathtt{clinical\_trial\_progresses}$, one $\exists C'.\ C \step C'$ per oracle/projection redex named in Figure~\ref{fig:clinical-trial}), and a \emph{sequential} statement chaining the redexes through a single trace ($\mathtt{ClinicalTrial.clinicalTrialPrefix\_steps}$ -- a 7-step $\mathtt{Steps}$ chain through Schema-gen, Policy-gen, Array[Schema]-gen, and the cost projection -- both in \texttt{Examples.lean}). The independent form is the natural progress lemma; the sequential form is the regression test that a single oracle/policy environment sustains the full prefix.

\paragraph{The \texttt{js}-projection boundary.} Field access in surface \hadl is delegated to JavaScript: \texttt{js visit.cost} and \texttt{js visit.patient\_id} in Figure~\ref{fig:clinical-trial} are JS expressions that read fields off the materialized record through the Boa bridge. The Lean mechanization models this boundary internally with $\mathtt{Expr.proj}$ and the rules $\rulename{Proj\text{-}Cong}$ / $\rulename{Proj\text{-}Step}$: the surface grammar has no first-class \texttt{e.f} form, but the operational story of ``a record is well-formed iff every field read on it succeeds'' must be expressible to state the self-heal trigger. $\rulename{Proj\text{-}Step}$ is therefore the mechanization-only model of a $\mathtt{js}$-escape that performs a lookup; it carries the witness $\mathtt{fs.lookup f = some\ v}$ that grounds projection success (the field-witness G1).

\paragraph{T4 corollaries.} The headline progress lemma (T4b, T4c) is parametric over $\mathrm{Healable}\ \tau$. The four shape corollaries -- $\mathtt{T4\_truthful\_success}$ ($\tau = \Tschema$), $\mathtt{T4\_truthful\_success\_arrow}$ ($\tau = \Tarrow{T_1,\ldots,T_n}{T'}$), $\mathtt{T4\_progress\_gen\_policy}$ ($\tau = \Tpolicy$), and the $\Tarr{\Tschema}$ instance proved in $\mathtt{nested\_array\_of\_schema\_succeeds}$ -- are 1-line specializations of the parametric $\mathtt{T4\_truthful\_success\_healable}$ in \texttt{Safety.lean}.

\paragraph{Forward field-access analysis.} The runtime refinement of the static continuation type-check is the syntactic analysis $\mathtt{Expr.fieldSitesAt} : \mathtt{Expr} \to \mathtt{Nat} \to \mathtt{List\ String}$ (\texttt{Syntax.lean}). At each \texttt{let}-gen site the analysis collects every projection on the bound variable in the continuation; the materialized record must satisfy $\mathtt{Value.recordSatisfies}$ on the resulting list, otherwise $\rulename{Let\text{-}Gen\text{-}Heal\text{-}RecordFields}$ fires. The analysis is over-approximate (no path-sensitivity, no element-field tracking inside \texttt{forE} over arrays of records) and is sound by construction: success is admitted only when every \emph{syntactically reachable} forward projection is satisfied.
