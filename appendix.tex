\section{Threat Model and Out-of-Scope Concerns}
\label{app:threat}

\paragraph{Trust boundaries.} \hadl treats the programmer-authored program text, the static type checker, the runtime re-checker, and the installed Cedar decision procedure as trusted. It treats every LLM output, whether a value, a schema, a workflow AST, or a policy, as untrusted. The retry budget $\MRetry$ declared by \texttt{@retries} is a bound on how many untrusted attempts the runtime will consume before failing closed on a given generation site.

\paragraph{What is caught.} Validation rejects an LLM output that fails the structural checks of the annotated type: a schema whose $\sigma$-image is not representable, a workflow AST that fails to type-check at its use site, a policy that triggers the deny-all guard or references an unknown action type. Every \texttt{gen} and \texttt{agent} invocation is checked against the installed policy before the LLM is called, under the explicit principal named by the surrounding \texttt{as} clause; a denial at this check becomes an $\ErrEv{\cdot}$ event and triggers retry. Append-only merging guarantees that no \texttt{enforce} step can relax an already-installed restriction.

\paragraph{What is not caught.} Cedar authorizes at the granularity of a \texttt{(principal, action, resource)} triplet where the action is drawn from the fixed set $\{\texttt{gen},\texttt{agent}\}$ and the resource is the call site. It does not model the downstream side effects of an authorized \texttt{agent} call, the contents of its prompt, or the data returned by the LLM. Four categories of threat are therefore outside the current enforcement model. First, \emph{prompt injection} against the LLM itself: a \texttt{gen} call whose prompt is composed with untrusted \texttt{ask} input can still be steered by an adversary who controls the user. Second, \emph{semantic misalignment} of generated policies: the deny-all guard and action-type check filter out policies that are syntactically pathological, but an LLM-generated policy that is merely \emph{over-permissive} with respect to operator intent will install and appear well-formed. Third, \emph{data exfiltration} via \texttt{say}, where an already-authorized workflow emits sensitive values to the human. Fourth, \emph{side-effecting js blocks}, which execute arbitrary JavaScript under the host runtime's ambient authority. These gaps are out of scope for gradual validation as we define it; the append-only discipline prevents policy relaxation but cannot recover from a malicious or inappropriate \emph{initial} policy.

\section{Type-System Limitations}
\label{app:limitations}

The type system deliberately makes several simplifications to keep gradual validation tractable.

\begin{itemize}
  \item \textbf{Arrays are homogeneously element-typed.} \texttt{array[T]} requires every element to have type $T$; there are no tuple types and no heterogeneous arrays.
  \item \textbf{Records check declared fields only.} A record type $\{\overline{x_i{:}T_i}\}$ requires every listed field to be present with the listed type, but the runtime value may carry additional fields. There is no row polymorphism, no width subtyping beyond the \texttt{Schema}-as-top rule, and no structural subtyping between record types.
  \item \textbf{No unions or intersections.} There are no sum types, union types, or refinement types. Values whose shape depends on runtime choice must flow through \texttt{Schema} and be re-checked at use.
  \item \textbf{\texttt{Schema} is top during static checking.} Both $T \le \texttt{Schema}$ and $\texttt{Schema} \le T$ hold for every $T$, so a \texttt{Schema}-typed value passes every static field-access and \texttt{eval} check; field-level validity is deferred to the post-materialization re-checker and, for compound schemas that $\sigma$ maps back to \texttt{Schema}, to the JavaScript engine at runtime.
  \item \textbf{\texttt{js} blocks are oracular.} Inline \texttt{js} expressions produce a value whose runtime type is whatever \texttt{RtType} accepts; the \hadl type checker does not model the JS type lattice.
  \item \textbf{No parametric polymorphism.} Workflow signatures are monomorphic.
\end{itemize}

\section{Evaluation Methodology}
\label{app:eval-method}

\paragraph{Workflow corpus.} The six workflows are the clinical-trial task of \figr{fig:clinical-trial}, the car scorer of \sect{sec:overview}, and four additional tasks: a customer-support triage workflow (\texttt{Schema}, \texttt{Policy}), a code-review agent (arrow type, \texttt{Policy}), a financial-compliance report (\texttt{Schema}, \texttt{Policy}), and a research-paper summarizer (\texttt{Schema}).

\paragraph{Models.} Each workflow is run against three LLMs: GPT-4.1, Claude-Sonnet-4, and the default model exposed by the GitHub Copilot SDK at the time of evaluation. All three are accessed through the same \hadl oracle interface.

\paragraph{Runs.} Each (workflow, model) pair is executed $K$ times with distinct seeds and distinct synthetic user inputs drawn from a per-workflow input template, for a total of $6 \times 3 \times K$ runs. Workflows that depend on human input (\texttt{ask}) are driven by a scripted stub.

\paragraph{Retry budget.} All experiments fix $\MRetry = 3$, matching the \texttt{@retries 3} pragma on Figure~\ref{fig:clinical-trial}.

\paragraph{Error taxonomy.} Every terminating run is classified as one of \emph{success} (reduction reaches a value configuration); \emph{type rejection} ($\rulename{Oracle-Heal-Type}$ consumed the full retry budget at some \texttt{gen} site); \emph{policy rejection} ($\rulename{Oracle-Heal-Pol}$ consumed the full budget); \emph{policy-install failure} (deny-all guard or action-type check rejected an \texttt{enforce} policy after $\MRetry$ retries); or \emph{runtime error} (non-oracle exception: JS engine failure, network error).

\paragraph{Determinism controls.} For each (workflow, model) pair we record the full prompt sequence and the full oracle response trace. This lets us replay a failure deterministically for the post-mortem and makes the token-cost numbers reproducible.

\section{Baseline Comparison Details}
\label{app:baselines}

Qualitative comparison between \hadl and the three baselines on the clinical-trial workflow.

\begin{description}
  \item[Language-level type discipline.] \hadl enforces structural types on \texttt{gen} outputs at declaration sites and rechecks after materialization. LangGraph carries no type information between nodes beyond programmer-authored Python annotations. LMQL constrains the \emph{shape} of a single LLM output at decoding time but does not track the type across calls. DSPy's signature syntax is the closest in spirit but operates at the module level, not the workflow level.
  \item[Per-invocation authorization.] \hadl's \texttt{enforce} + \texttt{as}-clause pattern produces a Cedar check at every \texttt{gen}/\texttt{agent} call site. In the three baselines, authorization is either at the process boundary or authored by hand in the node body; no baseline supports append-only policy merging.
  \item[Self-healing discipline.] \hadl's retry is driven by the violated constraint itself (type shape, Cedar denial) which is written back into the prompt. DSPy includes a feedback mechanism through its optimizer, but it operates at training time rather than per-invocation. LangGraph and LMQL have no built-in typed retry.
  \item[Formal guarantees.] \hadl has a mechanized per-step soundness (\sect{sec:sem-correctness}). The baselines have no formal model.
\end{description}

\section{Full Reduction Rules}
\label{app:rules}

This appendix lists the reduction rules elided from \sect{sec:sem-reduction} --- the remaining pure-core root rules, the ask/say rules, closure application, and the structural congruence rules --- followed by the proofs of Theorems~\ref{thm:hadl-soundness}, \ref{thm:hadl-progress}, and \ref{thm:hadl-term}. Left-to-right call-by-value evaluation order is enforced throughout by the structural congruence rules. The substitution $\Inst{e}{v}$ denotes the capture-avoiding instantiation of de-Bruijn index $0$ in $e$ with the value $v$ (decrementing all other free indices); for a closure body with $n$ binders, simultaneous substitution $e[v_0/0,\dots,v_{n-1}/(n{-}1)]$ is used.

\subsection{Let-Bind, Enforce-Install, and the Oracle Rules}
\label{app:rules-oracle}

\begin{mathpar}
\inferrule*[right=\rulename{Let-Bind}]
  {\mathsf{isValue}(v) \\ \RtType{v}{\tau}}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{v}{e}} \step \MConfig{\MErr}{\MPol}{\Inst{e}{v}}}

\inferrule*[right=\rulename{Let-Cong}]
  {\MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfig{\MErr}{\MPol}{\Let{\tau}{e_1}{e_2}} \step \MConfig{\MErr'}{\MPol'}{\Let{\tau}{e_1'}{e_2}}}

\inferrule*[right=\rulename{Enforce-Install}]
  {\Install{\MPol}{p} = \MPol'}
  {\MConfig{\MErr}{\MPol}{\Enforce{p}} \step \MConfig{\MErr}{\MPol'}{\Tunit}}

\inferrule*[right=\rulename{Oracle-Heal-Pol}]
  {\lnot\,\PolOk{\MPol}{a.\MPrin}{a.\mathsf{eff}} \\
   \epsilon = \Explain{a.\MPrin}{\MPol} \\
   \retries{\MErr \cdot \ErrEv{\epsilon}} \le \MRetry}
  {\MConfig{\MErr}{\MPol}{a.e} \step \MConfig{\MErr \cdot \ErrEv{\epsilon}}{\MPol}{a.e}}
\end{mathpar}

$\rulename{Oracle-Success}$ and $\rulename{Oracle-Heal-Type}$ appear in \figr{fig:rules-core}. The action tag $a$ projects $a.s$ (statement), $a.\tau$ (expected type, $\Tstring$ for agent), $a.\MPrin$ (principal), $a.\mathsf{eff} \in \{\texttt{gen},\texttt{agent}\}$, and $a.e$ (expression head). The budget metric $\retries{\MErr}$ is the length of the trailing $\ErrEv{\cdot}$ run since the most recent $\SucEv$, so a successful oracle step resets it to zero. $\Install{\MPol}{p}$ bundles the deny-all guard, action-type validation, and append-only merge; $\mathtt{policyInstall\_shrinks}$ guarantees $\Allow{\MPol'} \subseteq \Allow{\MPol}$.

\subsection{Ask, Say, and Closure Application}
\label{app:rules-io}

\begin{mathpar}
\inferrule*[right=\rulename{Ask}]
  {\Oracle{s}{\MErr}{\Tstring} \ni v \\
   \mathsf{isValue}(v) \\
   \RtType{v}{\Tstring}}
  {\MConfig{\MErr}{\MPol}{\Ask{s}} \step \MConfig{\MErr \cdot \SucEv}{\MPol}{v}}

\inferrule*[right=\rulename{Say}]
  { }
  {\MConfig{\MErr}{\MPol}{\Say{s}} \step \MConfig{\MErr}{\MPol}{\Tunit}}

\inferrule*[right=\rulename{Eval-Success}]
  {|\overline{v_i}| = n \\ \forall i.\ \mathsf{isValue}(v_i)}
  {\MConfig{\MErr}{\MPol}{\Eval{(\Clos{n}{e_b})}{\overline{v_i}}} \step \MConfig{\MErr}{\MPol}{e_b[v_0/0,\dots,v_{n-1}/(n{-}1)]}}
\end{mathpar}

$\rulename{Eval-Success}$ applies a closure $\Clos{n}{e_b}$ to a list of $n$ values by simultaneous substitution. Unlike the environment-based formulation, there is no environment extension and no per-argument $\RtType{\cdot}{\cdot}$ side condition at the application site: each $v_i$ was produced by an earlier reduction step whose own rule carried the runtime-type obligation (T2 then guarantees every value has \emph{some} runtime type; the black-boxed static checker handles shape agreement). Lifting $\rulename{Eval-Success}$ to non-root positions is standard; the function position of $\Eval{f}{\overline{v_i}}$ is driven to a closure value by $\rulename{Eval-Fun-Cong}$ below.

\subsection{Structural Congruence}
\label{app:rules-cong}

\begin{mathpar}
\inferrule*[right=\rulename{If-Cong}]
  {\MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfig{\MErr}{\MPol}{\Iff{e_1}{e_2}{e_3}} \step \MConfig{\MErr'}{\MPol'}{\Iff{e_1'}{e_2}{e_3}}}

\inferrule*[right=\rulename{Seq-Cong}]
  {\MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfig{\MErr}{\MPol}{\Seq{e_1}{e_2}} \step \MConfig{\MErr'}{\MPol'}{\Seq{e_1'}{e_2}}}

\inferrule*[right=\rulename{For-Cong}]
  {\MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfig{\MErr}{\MPol}{\Forr{e_1}{e_2}} \step \MConfig{\MErr'}{\MPol'}{\Forr{e_1'}{e_2}}}

\inferrule*[right=\rulename{Enforce-Cong}]
  {\MConfig{\MErr}{\MPol}{e} \step \MConfig{\MErr'}{\MPol'}{e'}}
  {\MConfig{\MErr}{\MPol}{\Enforce{e}} \step \MConfig{\MErr'}{\MPol'}{\Enforce{e'}}}

\inferrule*[right=\rulename{Eval-Fun-Cong}]
  {\MConfig{\MErr}{\MPol}{f} \step \MConfig{\MErr'}{\MPol'}{f'}}
  {\MConfig{\MErr}{\MPol}{\Eval{f}{\overline{v_i}}} \step \MConfig{\MErr'}{\MPol'}{\Eval{f'}{\overline{v_i}}}}
\end{mathpar}

None of the congruence rules carries a $\neq \mathsf{err}$ side condition: the language has no $\mathsf{errTerm}$ constructor, so a stuck configuration has no outgoing transition and its stuckness simply propagates up by absence.

There is no separate heal rule for $\Enforce{\cdot}$, for $\Agent{\cdot}{\cdot}$, or for closure application: the unified $\rulename{Oracle-Heal-Type}$ and $\rulename{Oracle-Heal-Pol}$ rules in \figr{fig:rules-core} subsume all gen/agent self-healing via the $\mathtt{OAction}$ action tag, and an $\Enforce{\cdot}$ whose policy value fails to install simply gets stuck --- there is no rewind-to-gen fallback in the mechanized semantics. Earlier drafts of the formalization included an $\rulename{Enforce-Heal}$ rule; it has been removed together with the $\mathsf{errTerm}$ constructor that it targeted.

\subsection{Mutable-State Rules}
\label{app:rules-mut}

The following rules are the only ones that read or write the store $\MStore$; all other rules in \sect{app:rules} leave $\MStore$ unchanged and elide it.

\begin{mathpar}
\inferrule*[right=\rulename{VarDecl-Cong}]
  {\MConfig{\MErr}{\MPol}{e_1} \step \MConfig{\MErr'}{\MPol'}{e_1'}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{var}\;x{:}\tau = e_1\;\mathtt{in}\;e_2} \step \MConfigS{\MErr'}{\MPol'}{\MStore}{\mathtt{var}\;x{:}\tau = e_1'\;\mathtt{in}\;e_2}}

\inferrule*[right=\rulename{VarDecl-Bind}]
  {\mathsf{isValue}(v) \\ \RtType{v}{\tau}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{var}\;x{:}\tau = v\;\mathtt{in}\;e_2} \step \MConfigS{\MErr}{\MPol}{\MStore[x \mapsto (\tau,v)]}{e_2}}

\inferrule*[right=\rulename{Assign-Cong}]
  {\MConfig{\MErr}{\MPol}{e} \step \MConfig{\MErr'}{\MPol'}{e'}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x \mathrel{:=} e} \step \MConfigS{\MErr'}{\MPol'}{\MStore}{x \mathrel{:=} e'}}

\inferrule*[right=\rulename{Assign-Write}]
  {\MStore(x) = (\tau, v_{\mathit{old}}) \\ \mathsf{isValue}(v) \\ \RtType{v}{\tau}}
  {\MConfigS{\MErr}{\MPol}{\MStore}{x \mathrel{:=} v} \step \MConfigS{\MErr}{\MPol}{\MStore[x \mapsto (\tau,v)]}{\Tunit}}

\inferrule*[right=\rulename{VarRead}]
  {\MStore(x) = (\tau, v)}
  {\MConfigS{\MErr}{\MPol}{\MStore}{\mathtt{read}\;x} \step \MConfigS{\MErr}{\MPol}{\MStore}{v}}
\end{mathpar}

$\rulename{VarDecl-Bind}$ and $\rulename{Assign-Write}$ carry the runtime-type side condition $\RtType{v}{\tau}$, so the store invariant $\forall x,\tau,v.\; \MStore(x) = (\tau, v) \Rightarrow \RtType{v}{\tau}$ (a conjunct of $\WF{\cdot}$) is preserved by every write. $\rulename{VarRead}$ resolves a read through the store and cannot be stuck on a well-formed configuration where the $x$ was declared. Mutable variables are \emph{not} de-Bruijn indices: they use named keys in $\MStore$ and never interact with the capture-avoiding substitution underlying \texttt{let} and closure application.

\section{Proofs of Theorems~\ref{thm:hadl-soundness} and \ref{thm:hadl-progress}}
\label{app:proofs}

Each proof is an induction on the derivation of $C \step C'$, mirroring the Lean case split on the constructor of $\mathtt{Step}$. The mechanization is free of user-introduced axioms: $\mathtt{policyInstall\_shrinks}$ is proved by iterated application of Cedar-Lean's $\mathtt{unchanged\_deny\_when\_add\_forbid}$ theorem, and the JavaScript host evaluator is modelled by an opaque $\mathtt{jsEval} : \mathtt{JsExpr} \to \mathsf{Option}\;\mathtt{Value}$ whose stub implementation returns $\mathsf{none}$; consequently the Lean $\mathtt{jsEval\_wellTyped}$ lemma is vacuous, and the paper-level ``oracular \texttt{js}'' story is not yet axiomatized --- a faithful axiomatization would replace the stub with a single $\mathtt{js\_oracular}$ axiom and add it to the allowlist checked by $\mathtt{AxiomCheck.lean}$. The policy-install operation $\Install{\cdot}{\cdot}$ is \emph{partial} (Lean's $\mathtt{policyInstall}$ returns $\mathsf{Option}$); the paper's $\MPol' = \Install{\MPol}{p}$ notation is shorthand for ``on the branch where installation succeeds''. All theorems depend only on Lean's standard meta-axioms ($\mathtt{propext}$, $\mathtt{Classical.choice}$, $\mathtt{Quot.sound}$); no other steps are admitted.

\subsection{T1: WF-preservation}
\label{app:proof-sound-a}

The invariant is the single clause $\retries{\MErr} \le \MRetry$. The proof is by cases on the $\mathtt{Step}$ constructor in \texttt{Soundness.lean}; the non-trivial cases are:

\paragraph{Case $\rulename{Ask}$, $\rulename{Oracle-Success}$.} The target heal context is $\MErr \cdot \SucEv$; by the trailing-success lemma $\mathtt{ErrCtx.retries\_append\_success}$ its retry metric is $0 \le \MRetry$.

\paragraph{Case $\rulename{Oracle-Heal-Type}$, $\rulename{Oracle-Heal-Pol}$.} The rule's own premise $\retries{\MErr \cdot \ErrEv{\epsilon}} \le \MRetry$ is exactly the required invariant on $C'$.

\paragraph{Pure-core and congruence cases.} $\MErr$ is either unchanged or inherits $\le \MRetry$ from the inductive hypothesis on the inner step.

Because $\WF{C}$ no longer involves environment well-typedness or residual static typeability, the per-rule cases are uniformly trivial --- this is the mechanized content of $\mathtt{T1\_WF\_preservation}$.

\subsection{T2: Staged materialization}
\label{app:proof-sound-b}

If the step $C \step C'$ produces a value $v$ at $C'$'s expression slot, then $\exists \tau.\ \RtType{v}{\tau}$. The proof ($\mathtt{T2\_staged\_materialization}$) factors through a value-lemma ``$\mathsf{isValue}(v) \Rightarrow \exists \tau.\ \RtType{v}{\tau}$'' that enumerates the value constructors. For base literals and schema/policy values the corresponding $\mathtt{vUnit}/\mathtt{vBool}/\mathtt{vInt}/\mathtt{vStr}/\mathtt{vSchema}/\mathtt{vPol}$ case applies. For closures, the $\mathtt{vClos}$ rule admits an arrow of matching arity; the T2 witness instantiates arguments and return to $\Tunit$, giving $\tau = \Tarrow{\overline{\Tunit}}{\Tunit}$ --- the precise argument and return types of the body are irrelevant to T2's existential claim. For record and array values, $\mathtt{vRec}$ and $\mathtt{vArr}$ instantiate witnesses at records with an empty field list and arrays of $\Tunit$, again as existential witnesses. This is strictly stronger than the prior ``materialization correctness'' claim, which only covered values returned by $\rulename{Gen-Success}$ at the annotated type; the new statement applies uniformly at every reduced expression and removes the need for the staged-materialization sub-argument in downstream proofs.

\subsection{T3: Policy monotonicity}
\label{app:proof-sound-c}

$\MPol$ is updated only by $\rulename{Enforce-Install}$, whose target policy $\MPol' = \Install{\MPol}{p}$ satisfies $\Allow{\MPol'} \subseteq \Allow{\MPol}$ by $\mathtt{policyInstall\_shrinks}$. All other rules leave $\MPol$ unchanged and give $\Allow{\MPol'} = \Allow{\MPol}$, hence the reflexive subset. The pointwise authorization claim is immediate because $\rulename{Oracle-Success}$, $\rulename{Oracle-Heal-Type}$ all carry $\PolOk{\MPol}{a.\MPrin}{a.\mathsf{eff}}$ as a side condition (and $\rulename{Oracle-Heal-Pol}$ carries its negation, which is why it exists).

\subsection{T4a: Budget → no heal}
\label{app:proof-prog-a}

Assume $C = \MConfig{\MErr}{\MPol}{a.e}$ with $\retries{\MErr} > \MRetry$ and $C \step C'$. We case-split on the $\mathtt{Step}$ constructor producing $C'$. Every pure-core, congruence, and $\mathtt{enforceInstall}/\mathtt{evalSuccess}/\mathtt{letBind}/\ldots$ case immediately contradicts $e = a.e$ by constructor distinctness. The $\rulename{Oracle-Heal-Type}$ and $\rulename{Oracle-Heal-Pol}$ cases have $\retries{\MErr \cdot \ErrEv{\epsilon}} \le \MRetry$ as a premise; rewriting with $\mathtt{ErrCtx.retries\_append\_error}$ yields $\retries{\MErr} + 1 \le \MRetry$, contradicting the hypothesis $\retries{\MErr} > \MRetry$ by $\mathtt{omega}$. The only surviving case is $\rulename{Oracle-Success}$, which indeed transitions to $\MConfig{\MErr \cdot \SucEv}{\MPol}{v}$. This is mechanized as $\mathtt{T4\_budget\_no\_heal}$ in \texttt{Safety.lean}.

Notably, T4a does not claim that $C$ is \emph{stuck} --- an oracle that is still truthful at the exhausted heal context can still fire $\rulename{Oracle-Success}$. The claim is only that no \emph{heal} rule applies, which is the semantically meaningful part: a budget-exhausted configuration cannot accumulate more feedback events.

\subsection{T4b: Truthful success}
\label{app:proof-prog-b}

Fix an oracle redex $a.e$ (either $\Gen{\tau}{s}{\MPrin}$ or $\Agent{s}{\MPrin}$) with $\PolOk{\MPol}{\MPrin}{a.\mathsf{eff}}$. By \defn{def:oracle-truthful} there exists some $\MErr'$ with $\retries{\MErr'} \le \MRetry$ and a value $v$ with $\Oracle{s}{\MErr'}{\tau} \ni v$, $\mathsf{isValue}(v)$, and $\RtType{v}{\tau}$. Apply $\rulename{Oracle-Success}$ at the action tag $a$: all four side conditions are discharged, so $\MConfig{\MErr'}{\MPol}{a.e} \step \MConfig{\MErr' \cdot \SucEv}{\MPol}{v}$. Mechanized as $\mathtt{T4\_truthful\_success}$ (the agent variant is $\mathtt{T4\_truthful\_success\_agent}$; it uses a direct oracle witness at the caller's $\MErr$ because the $\mathtt{eventuallyTruthful}$ predicate returns only an unspecified heal context).

\subsection{T4c: Progress-gen, Progress-agent}
\label{app:proof-prog-c}

These are wrappers around T4b: they add $\PolOk{\MPol}{\MPrin}{a.\mathsf{eff}}$ as an explicit hypothesis and take a direct oracle witness $\exists v.\ \mathsf{isValue}(v) \wedge \RtType{v}{\tau} \wedge \Oracle{s}{\MErr}{\tau} \ni v$ at the caller's \emph{current} heal context. The conclusion $\exists C'.\ C \step C'$ follows by applying $\rulename{Oracle-Success}$ exactly as in T4b but without the intermediate ``reach some $\MErr'$'' step. Mechanized as $\mathtt{T4\_progress\_gen}$ / $\mathtt{T4\_progress\_agent}$ in \texttt{Safety2.lean}.

\subsection{Termination}
\label{app:term}

\begin{theorem}[Termination]
\label{thm:hadl-term}
Let $C_0 = \MConfig{\emptyset}{\MPol_0}{e_0}$ be a well-formed initial configuration. Assume every \texttt{while}/\texttt{for} loop in $e_0$ terminates. Then every trace from $C_0$ either reaches a value configuration in finitely many steps or gets stuck at some oracle redex whose retry budget has been exhausted without a truthful response --- with no ill-formed intermediate configuration and no terminal-error marker. If $\mathcal{O}$ is eventually-truthful at every oracle site encountered along the trace, the trace terminates in a value configuration.
\end{theorem}

\label{app:proof-term}
The argument is on paper. Between two successive oracle firings, only pure-core and congruence rules fire. Pure-core reduction terminates on any expression whose $\texttt{while}$ and $\texttt{for}$ loops terminate (assumed of $e_0$); substitution is finitary and preserves the termination measure on loop-free code. At each oracle firing either $\SucEv$ is appended (resetting $\retries{\cdot}$ to $0$ by T1) or an $\ErrEv{\cdot}$ extends the trailing retry run by one. After at most $\MRetry + 1$ heals the configuration is either pushed into the ``budget exceeded'' regime of T4a --- where only $\rulename{Oracle-Success}$ can fire, so the configuration is either truthful-success-progressing or stuck by absence of transition --- or has already succeeded. Under eventually-truthful $\mathcal{O}$, truthful-success is guaranteed at every site; since the number of oracle sites reached along the trace is finite (bounded by $|e_0|$ plus finitely many $\rulename{Eval-Success}$ expansions), the trace terminates in a value configuration. Otherwise the trace either terminates in a value or gets stuck at an oracle redex for which no witness is ever produced; no ill-formed configuration, and no $\mathsf{errTerm}$ marker, is ever reached, since neither exists in the language.
