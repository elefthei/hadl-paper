\section{Implementation}
\label{sec:implementation}

This section describes the compilation pipeline, runtime execution model, and authorization integration of the HADL implementation. The system is implemented in Rust as a workspace of six crates excluding the vendored Copilot SDK dependency.

\subsection{Compilation Pipeline}
\label{sec:impl-pipeline}

HADL programs are processed in three phases, with no intermediate representation or code generation step between them.

\paragraph{Phase 1. Parsing.} The parser is generated from a PEG grammar of 54 rules using the \texttt{pest} parser generator. The grammar defines whitespace as spaces and tabs (not newlines), treats newlines as expression separators within sequences, and reserves 23 keywords. The parser transforms source text into a typed AST whose nodes correspond directly to the \texttt{Expr}, \texttt{Type}, and \texttt{Declaration} types defined in Section~\ref{sec:design-syntax}. String literals are further processed to extract backtick-delimited interpolation segments, which are stored as \texttt{StringPart::JsExpr} nodes for evaluation by the JavaScript engine at runtime.

\paragraph{Phase 2. Type checking.} The type checker implements a two-pass algorithm. The first pass collects all workflow and type declarations, building curried arrow types for workflows and registering type aliases. The second pass checks each workflow body against its declared return type using type inference with subsumption. Arrow subtyping is contravariant in the parameter and covariant in the return type. \texttt{tool} and \texttt{js} expressions are treated as oracles that can produce any checked type, requiring type annotations at binding sites. All type errors carry source spans for diagnostic reporting. The type checker is also invoked at runtime during staged re-typechecking, as described in Section~\ref{sec:design-gradual-types}.

\paragraph{Phase 3. Interpretation.} The interpreter walks the AST directly, evaluating each expression node recursively. There is no compilation to bytecode or intermediate representation. The interpreter is generic over a \texttt{UserComm} trait that abstracts human interaction, allowing the same interpreter to run with standard input in headless mode or with an interactive TUI dashboard. The interpreter communicates with the dashboard through an unbounded channel of \texttt{RuntimeEvent} values that carry source spans, tool descriptions, streaming LLM output deltas, and completion status.

\subsection{Runtime Execution Model}
\label{sec:impl-runtime}

The interpreter maintains five pieces of state that together support the three forms of gradual generation, provenance tracking, and policy enforcement.

\paragraph{Unified environment.} Each variable binding is stored as a struct containing the runtime JSON value, a mutability flag, an optional concrete schema type, optional provenance information, and a policy flag. This unified representation keeps the value, its type, its causal history, and its role in gradual generation synchronized.

\paragraph{JavaScript engine.} An embedded JavaScript engine (Boa) with a persistent context handles all arithmetic, property access, and backtick interpolation. HADL variables are injected into the JavaScript global scope before each evaluation, and mutable variables are read back afterward.

\paragraph{Expression evaluation.} The interpreter is a recursive async function that pattern-matches on expression variants. Bindings record provenance for \texttt{tool} and \texttt{agent} calls and detect \texttt{Schema} and \texttt{Policy} annotations to trigger gradual generation. Sequencing evaluates left-to-right, checking after each expression whether staged re-checking or policy installation should run.

\paragraph{Staged re-checking and self-healing.} The sequence evaluator is the site where staged gradual type checking executes at runtime. After each \texttt{Schema}-annotated binding, the interpreter constructs a \texttt{TypeEnv} from the current runtime state and re-invokes the type checker over the remaining expressions. An initial best-effort re-check discards errors (since subsequent expressions may mutate the schema), but when a schema-typed variable is reassigned (indicated by a \texttt{schema\_dirty} flag), an authoritative re-check is performed. If this fails and the \texttt{@retries} pragma is active, provenance tracking identifies the causal tool call by following dependency chains transitively. Each variable in the chain is retried with the type error appended to the original prompt, up to the declared retry limit.

\paragraph{Gradual policy installation.} The \texttt{enforce} expression triggers the installation pipeline described in Section~\ref{sec:design-gradual-policies}. The interpreter verifies the policy flag on the target variable, runs the deny-all guard and action type validation, deserializes and merges the policy append-only, compiles the merged specification into Cedar via a new \texttt{CedarValidator}, and performs scoped principal validation. Compilation failures invoke provenance-based self-healing. Once installed, every subsequent \texttt{tool} and \texttt{agent} invocation is validated against the merged policy before the LLM call is made.

\paragraph{Workflow invocation and LLM integration.} The \texttt{eval} expression deserializes a JSON value as a \texttt{Program} AST, validates arguments against the workflow's parameters, and evaluates the body with closure-like semantics. The interpreter communicates with LLMs through the GitHub Copilot SDK, which maintains separate tool and agent sessions with headless and streaming modes. Responses are parsed by attempting JSON deserialization, markdown code block extraction, and finally falling back to a plain string.

\subsection{Authorization Integration}
\label{sec:impl-auth}

Policies are defined in JSON files specifying permit and forbid rules with principal, action, resource, and condition patterns. At startup, each rule is translated into Cedar syntax: principal node-type constraints become Cedar \texttt{when} clauses, and \texttt{file\_matches} conditions become Cedar \texttt{like} operators. Before each \texttt{tool} and \texttt{agent} invocation, the interpreter constructs a (principal, action, resource) triplet from the call-site counter and the \texttt{as} clause (defaulting to \texttt{"tool"} or \texttt{"agent"}), and passes it to the Cedar \texttt{Authorizer}. A deny decision halts the workflow before the LLM call is made.
