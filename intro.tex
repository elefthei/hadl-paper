\section{Introduction}
\label{sec:intro}

Modern agents write substantial programs. A single agent run now sequences many gen calls, branches on intermediate results, accumulates mutable state, and delegates subtasks to other agents. The volume of code an agent authors grows with every model generation, but safety does not. A workflow of ten gen calls receives no more type discipline than a one-shot prompt, and authorization, when enforced at all, is applied at the outer boundary rather than at each call. Gen outputs flow from one call to the next with no check that their shape matches what downstream code expects, and sensitive gens can be invoked from within a workflow with no evidence that the invocation was authorized. As agents are trusted with more consequential work such as filling forms, querying sensitive records, and generating other agents, safety must scale with the size of the workflow rather than lag behind it.

Today developers express these plans using orchestration frameworks such as LangChain~\cite{langchain2022} or ad-hoc Python scripts. These approaches treat plans as configurations of library objects: gen calls return untyped data, control flow is scattered across callback edges, structural errors surface only at runtime, and authorization is applied at the outer boundary rather than per invocation. Neighboring programming-language approaches such as LMQL~\cite{beurer2023prompting} and DSPy~\cite{khattab2023dspy} address parts of this problem but do not integrate a type system for gen outputs with per-invocation authorization. Yue et al.~\cite{yue2026statictemplatesdynamicruntime} identify the tension between expressivity and verifiability as a central open problem in this space.

Figure~\ref{fig:clinical-trial} illustrates gradual validation and per-invocation authorization in a single workflow. Two \texttt{gen} calls produce program structure (a type and a policy) that are activated on the very next lines. The schema \texttt{gen} produces a type that is immediately re-checked against the remainder of the program: if a field accessed later (\texttt{visit.cost}, \texttt{visit.patient\_id}) is absent from the generated schema, the runtime retries the schema \texttt{gen} with the violated constraint as feedback before any patient data is collected. Every subsequent \texttt{gen} call is then typed against the concrete schema and checked against the generated Cedar policy under an explicit principal. The same type discipline that governs ordinary field access such as \texttt{visit.patient\_id} governs the LLM output that produced the value.

We present \hadl (Hierarchical Agentic Dataflow Language), a strongly typed imperative DSL for composing long-running agentic workflows. \hadl provides four domain-specific primitives (\texttt{ask}, \texttt{say}, \texttt{gen}, \texttt{agent}) as first-class typed expressions in an imperative language with mutable state, loops, and nested workflow composition. The type system supports three forms of gradual validation: \texttt{Schema} lets a gen produce a type that resolves to a concrete type once the LLM delivers one. Arrow types let a gen produce a complete workflow invoked via \texttt{eval}. \texttt{Policy} lets a gen produce a Cedar~\cite{cutler2024cedar} policy activated via \texttt{enforce} with append-only merging.

\textbf{Contributions.} (1)~We design \hadl, an imperative DSL in which \texttt{ask}/\texttt{say}/\texttt{gen}/\texttt{agent} are first-class typed expressions with per-invocation Cedar authorization, implemented in Rust. (2)~We introduce \emph{gradual validation}: a staged checking discipline in which types, workflows, and authorization policies produced by LLMs at runtime are checked against their declared types at use time, under the same type system that governs the rest of the program. Unlike classical gradual typing~\cite{siek2006gradual,siek2015refined}, \hadl inserts no runtime casts and does not satisfy the gradual guarantee. The contribution is not a new type-theoretic foundation but the unification of three LLM-generation patterns (type, workflow, policy) under a single re-checking plus bounded-retry discipline. All three share a feedback-driven retry that reruns the causal generator when validation fails.
