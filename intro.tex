\section{Introduction}
\label{sec:intro}

Modern agents write substantial programs. A single agent run now sequences many gen calls, branches on intermediate results, accumulates mutable state, and delegates subtasks to other agents. The volume of code an agent authors grows with every model generation; what does not grow is safety. A workflow of ten gen calls receives no more type discipline than a one-shot prompt, and authorization, when enforced at all, is applied at the outer boundary rather than at each call. Gen outputs flow from one call to the next with no check that their shape matches what downstream code expects, and sensitive gens can be invoked from within a workflow with no evidence that the invocation was authorized. As agents are trusted with more consequential work such as filling forms, querying sensitive records, and generating other agents, safety must scale with the size of the workflow rather than lag behind it.

Today developers express these plans using orchestration frameworks such as LangChain~\cite{langchain2022} or ad-hoc Python scripts. These approaches treat plans as configurations of library objects: gen calls return untyped data, control flow is scattered across callback edges, structural errors surface only at runtime, and authorization is applied at the outer boundary rather than per invocation. Neighboring programming-language approaches such as LMQL~\cite{beurer2023prompting} and DSPy~\cite{khattab2023dspy} address parts of this problem but do not integrate a type system for gen outputs with per-invocation authorization. Yue et al.~\cite{yue2026statictemplatesdynamicruntime} identify the tension between expressivity and verifiability as a central open problem in this space.

\begin{lstlisting}[language=hadl,showstringspaces=false]
@retries 3

workflow clinical_trial() -> String {
  let trial = ask "Describe the clinical trial"
  let crf: Schema = gen "Generate a CRF schema for `trial`"
  let policy: Policy = gen "Coordinators cannot see
    unblinding data. Safety_board has full access."
  enforce policy
  let role = ask "Your role?"
  let ids: Schema = gen "List 3 patient IDs"
  var total: Number = 0
  for pid in ids {
    let visit: crf = gen "Generate visit for `pid` in `trial`" as role
    total = js total + 1
    say "`visit.patient_id` recorded"
  }
  say "Collected `total` visits"
  let sae = ask "Adverse event reported?"
  if (sae == "yes") {
    let safety: Policy = gen "Coordinators cannot see adverse events."
    enforce safety
    gen "Write an SAE narrative" as "safety_board"
  }
  "done"
}
\end{lstlisting}
\caption{A clinical trial workflow. Line~5 produces a \texttt{Schema}; line~6 a \texttt{Policy} activated by \texttt{enforce} on line~8. Every \texttt{gen} on lines~13--15 is typed against the generated schema and authorized against the installed policy under the \texttt{role} principal (line~9). When a serious adverse event occurs, line~20 generates a tighter policy and line~21 merges it append-only, so access rules only tighten. \texttt{@retries 3} enables self-healing: if validation rejects an LLM output, the runtime re-invokes the causal \texttt{gen} call with the violated constraint as feedback.}
\label{fig:clinical-trial}
\end{figure}

Figure~\ref{fig:clinical-trial} illustrates gradual validation and per-invocation authorization in a single workflow. Two \texttt{gen} calls produce program structure --- a type and a policy --- that are activated on the very next lines. Every subsequent call is typed against the generated schema and checked against the generated policy under an explicit principal. When new information arrives, a second policy is generated and activated, merged append-only with the first, so access rules only tighten. The same type discipline that governs ordinary field access such as \texttt{visit.patient\_id} governs the LLM output that produced the value.

We present \hadl (Hierarchical Agentic Dataflow Language), a strongly typed imperative DSL for composing long-running agentic workflows. \hadl provides four domain-specific primitives (\texttt{ask}, \texttt{say}, \texttt{gen}, \texttt{agent}) as first-class typed expressions in an imperative language with mutable state, loops, and nested workflow composition. The type system supports three forms of gradual validation: \texttt{Schema} lets a gen produce a type that resolves to a concrete type once the LLM delivers one; arrow types let a gen produce a complete workflow invoked via \texttt{eval}; \texttt{Policy} lets a gen produce a Cedar~\cite{cutler2024cedar} policy activated via \texttt{enforce} with append-only merging.

\textbf{Contributions.} (1)~We design \hadl, an imperative DSL in which \texttt{ask}/\texttt{say}/\texttt{gen}/\texttt{agent} are first-class typed expressions with per-invocation Cedar authorization, implemented in Rust. (2)~We introduce \emph{gradual validation}: a staged checking discipline in which types, workflows, and authorization policies produced by LLMs at runtime are checked against their declared types at use time, under the same type system that governs the rest of the program. Unlike classical gradual typing~\cite{siek2006gradual,siek2015refined}, \hadl inserts no runtime casts and does not satisfy the gradual guarantee; the contribution is not a new type-theoretic foundation but the unification of three LLM-generation patterns (type, workflow, policy) under a single re-checking plus bounded-retry discipline. All three share a feedback-driven retry that reruns the causal generator when validation fails.
