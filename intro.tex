\section{Introduction}
\label{sec:intro}

Large language models have moved beyond single-turn question answering into the role of autonomous agents that decompose complex tasks, invoke external tools, incorporate human feedback, and iteratively refine their outputs over extended interactions~\cite{wei2022chain,yao2022react,schick2023toolformer}. These capabilities compose into sophisticated multi-turn workflows including hierarchical task decomposition~\cite{yao2023tree}, self-reflective refinement loops~\cite{shinn2024reflexion}, and coordinated multi-agent systems~\cite{wu2024autogen,hong2023metagpt}. Surveys~\cite{chen2024survey,huang2024understanding,yue2026statictemplatesdynamicruntime} reveal that at their core, all of these systems execute structured \emph{plans} that branch on conditions, iterate over collections, accumulate mutable state, invoke external services, and compose partial results into final outputs.

Today, developers express these plans using orchestration frameworks such as LangGraph~\cite{langgraph2024} and LangChain~\cite{langchain2022}, or through ad-hoc scripts in Python. These approaches treat plans as \emph{configurations of library objects} rather than as programs in their own right, with four consequences: tool calls return untyped data, control flow is expressed through graph construction rather than imperative constructs, structural errors surface at runtime rather than before execution, and authorization is intractable at the per-invocation granularity that agentic systems require. Existing orchestration frameworks~\cite{wu2024autogen,hong2023metagpt}, programming language approaches~\cite{beurer2023prompting,khattab2023dspy,daunis2025declarative}, and durable execution platforms~\cite{temporal2024} each address parts of this problem but none provides a static type system for tool outputs, compile-time error reporting, or authorization integrated into the execution semantics. Yue et al.~\cite{yue2026statictemplatesdynamicruntime} identify the tension between expressivity and verifiability as a central open problem. To our knowledge, no existing system applies programming language techniques to agentic workflow specification with per-invocation policy enforcement.

Our central observation is that agentic workflows are \emph{programs}. If plans are programs, then type systems, static analysis, and authorization semantics should apply directly. A deeper observation is that the LLM is not merely a value producer but a generator of program structure. Tool calls can produce \emph{types} that govern subsequent data, \emph{workflows} that will be executed as subroutines, and \emph{policies} that restrict future operations. These artifacts do not exist at compile time. We address this through \emph{gradual generation}, a mechanism in which LLMs produce types, workflows, and authorization policies at runtime, and the runtime validates and installs them before execution continues. For types, \texttt{Schema} serves as a universally compatible type during static checking~\cite{siek2006gradual}, with staged re-typechecking when concrete types materialize. For workflows, arrow types enable LLMs to generate complete typed workflows executed via \texttt{eval}. For policies, \texttt{Policy} enables LLMs to generate Cedar~\cite{cutler2024cedar} policies activated via an explicit \texttt{enforce} expression that separates generation from activation, with safety checks and append-only merging enforced before every subsequent call. All three share provenance-driven self-healing that retries the causal tool call when validation fails.

Figure~\ref{fig:example} illustrates gradual type generation. The first \texttt{tool} call generates a schema at runtime; the runtime converts it to a concrete type and re-invokes the type checker. The second \texttt{tool} call uses the schema variable in type annotation position. The \texttt{agent} call accesses fields via backtick expressions evaluated by the embedded JavaScript engine.

We present HADL (Hierarchical Agent Dataflow Language), a strongly typed imperative domain-specific language for composing long-running agentic workflows. HADL provides three domain-specific primitives (\texttt{user}, \texttt{tool}, and \texttt{agent}) as first-class typed expressions in an imperative language with mutable state, unbounded loops, structured iteration, and nested workflow composition. The type system supports three forms of gradual generation: the \texttt{Schema} type for runtime type generation with staged re-typechecking, arrow types for runtime workflow generation via \texttt{eval}, and the \texttt{Policy} type for runtime authorization policy generation via \texttt{enforce} with safety checks and append-only Cedar~\cite{cutler2024cedar} policy merging. The language is implemented in Rust with PEG parsing, static type checking, and async tree-walking interpretation, integrating with the GitHub Copilot SDK for LLM execution.

We make the following contributions:

\begin{enumerate}
	\item We design HADL, an imperative DSL in which \texttt{user}, \texttt{tool}, and \texttt{agent} are first-class typed expressions with curried function types, providing natural imperative syntax for multi-turn agent orchestration. A type checker with source-span diagnostics catches structural errors before any LLM call is made. The language is implemented in Rust with a PEG parser, type checker, and async tree-walking interpreter.
	\item We introduce \emph{gradual generation}, a mechanism in which LLMs produce types, workflows, and authorization policies at runtime. \texttt{Schema} serves as a universally compatible type during static checking, with staged re-typechecking when concrete types materialize. Arrow types enable LLM-generated workflows executed via \texttt{eval}. \texttt{Policy} enables LLM-generated Cedar authorization policies activated via \texttt{enforce}, with safety checks (deny-all rejection, action type validation), scoped principal validation, and append-only merging. All three share provenance-driven self-healing.
\end{enumerate}
