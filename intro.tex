\section{Introduction}
\label{sec:intro}

Large language models have moved beyond single-turn question answering into the role of autonomous agents that decompose complex tasks, invoke external tools, incorporate human feedback, and iteratively refine their outputs over extended interactions~\cite{wei2022chain,yao2022react,schick2023toolformer}. Recent work has demonstrated that these agentic capabilities can be composed into sophisticated multi-turn workflows, including hierarchical task decomposition~\cite{yao2023tree}, self-reflective refinement loops~\cite{shinn2024reflexion}, and coordinated multi-agent systems~\cite{wu2024autogen,hong2023metagpt}. Surveys of the rapidly expanding landscape~\cite{chen2024survey,huang2024understanding} reveal a common structural pattern. At their core, all of these systems execute \emph{plans}, that is, structured sequences of steps that branch on conditions, iterate over collections, accumulate mutable state, invoke external services, and compose partial results into final outputs. As the complexity and scale of these plans grows, from simple tool-calling loops to nested pipelines spanning hundreds of agent interactions, the need for the same engineering discipline we apply to programs becomes pressing, namely type safety, static analysis, compositional design, and principled error reporting.

Today, developers express agentic plans using general-purpose orchestration frameworks such as LangGraph~\cite{langgraph2024} and LangChain~\cite{langchain2022}, or through ad-hoc scripts in Python. These approaches treat plans as \emph{configurations of library objects}, that is, graphs to wire, chains to compose, and callbacks to register, rather than as programs in their own right. Four consequences follow. First, tool calls return untyped data, so mismatches between the structure a developer expects from an LLM and the structure actually produced are caught only at runtime, often after expensive API calls have already been made. Second, control flow is expressed through graph construction or callback patterns rather than through natural imperative constructs such as loops, conditionals, and variable bindings, forcing developers to reason about state transitions in a graph rather than about the sequential logic of their plan. Third, structural errors in the workflow, including undefined variables, type mismatches, and incompatible data shapes flowing between stages, surface during execution rather than before it, defeating the purpose of having a structured workflow in the first place. Fourth, the same absence of formal structure that renders type errors invisible also renders authorization intractable at the granularity that agentic systems require. Agent systems today operate with coarse-grained access control, where an agent either has access to all available tools and data, or to none. There is no mechanism to enforce fine-grained policies \emph{per invocation}, specifying which data a particular tool call may consume, whether a value produced by one agent may flow to another, or whether human approval is required before a sensitive operation proceeds. Because existing frameworks have no representation of data flow or type structure, they cannot express, let alone enforce, such policies. Authorization is treated as an external concern, bolted on after the workflow is designed.

Existing systems address parts of this problem but leave the core gap open. LangGraph~\cite{langgraph2024} provides durable execution and human-in-the-loop interrupts but requires manual graph construction and offers no static type checking. DSPy~\cite{khattab2023dspy} compiles declarative language model pipelines into optimized prompting strategies but targets single-turn structured generation, not long-running stateful workflows with mutable state and unbounded iteration. LMQL~\cite{beurer2023prompting} brings programming language ideas to LLM interaction through a query language with type constraints, representing the closest precedent in the PL literature, but focuses on prompt-level scripting rather than multi-turn agent orchestration with control flow and state. A recent declarative DSL for agent workflows~\cite{daunis2025declarative} separates workflow specification from implementation but uses DAG-based configuration without a type system or imperative control flow, and cannot express the iterative refinement patterns that characterize long-running agent interactions. AutoGen~\cite{wu2024autogen} and MetaGPT~\cite{hong2023metagpt} enable multi-agent collaboration but lack formal type systems or compile-time error detection. Critically, none of these systems integrate authorization into the execution semantics. Policy enforcement, where it exists, is handled by external middleware or ad-hoc checks scattered through application code, disconnected from the workflow's type structure and data flow. To our knowledge, no existing system brings programming language theory, specifically bidirectional type checking, structural subtyping, and compositional semantics, to the problem of agentic workflow specification with per-invocation policy enforcement.

Our central observation is that agentic workflows are \emph{programs} in every meaningful sense. They bind variables, branch on conditions, iterate over collections, accumulate mutable state, call external services, and compose into larger units. If plans are programs, then the mature tools of programming language design, including type systems, static analysis, effect management, and authorization semantics, should apply directly. Three specific observations guide our design. First, LLM tool calls occupy a distinctive position in the type-theoretic landscape. They behave as \emph{oracles} in the epistemic sense, that is, black boxes that can produce any type at any call site, with the developer's annotation at the binding site as the sole source of type information. This leads naturally to a bidirectional type checking discipline~\cite{pierce2000local,dunfield2021bidirectional} in which tool-call expressions have no synthesis rule and \emph{must} appear in a checking context, encoding the epistemological uncertainty of LLM outputs directly into the type system. Second, the data flowing through agent workflows is naturally represented as structural records with set-of-fields semantics~\cite{cardelli1984semantics,remy1989type}, where different workflow stages care about different subsets of fields and width subtyping~\cite{pierce2002types} allows values to flow between stages without explicit conversion. Third, governance in agentic systems is not a deployment concern but an \emph{execution semantics} concern. When a tool call produces a value, the system must track who produced it and enforce where it may flow. When an agent invocation is about to execute, the runtime must validate whether the current principal is authorized to perform that action on that resource. This validation must happen \emph{before every call}, as part of the language's evaluation semantics, not as an external afterthought. Finally, treating workflows as programs yields an expressiveness benefit. Because HADL code can appear as the string output of an \texttt{agent} invocation, an outer workflow can delegate sub-task planning to an LLM that generates HADL code for subsequent parsing and execution. This \emph{architect pattern} demonstrates that the language is expressive enough to support hierarchical plan generation using its existing primitives, without requiring additional orchestration mechanisms.

We present HADL (Hierarchical Agent Dataflow Language), a strongly typed imperative domain-specific language for composing long-running agentic workflows. HADL provides three domain-specific primitives, namely \texttt{user}, \texttt{tool}, and \texttt{agent}, that reify human interaction, LLM tool invocation, and LLM reasoning as first-class typed expressions in an imperative language with mutable state, unbounded loops, structured iteration, and nested workflow composition. The type system implements bidirectional checking~\cite{pierce2000local,dunfield2021bidirectional} with structural record subtyping featuring width and depth subtyping and field-order independence~\cite{cardelli1984semantics,remy1989type}, where tool-call expressions require explicit type annotations at binding sites, formalizing the boundary between typed program logic and untyped LLM outputs. Workflows are typed, curried functions that compose naturally and support the extended, stateful execution patterns characteristic of real-world agent deployments. A Cedar-based~\cite{cutler2024cedar} policy layer integrates authorization triplets (principal, action, resource) directly into the interpreter loop. Before every \texttt{tool} and \texttt{agent} invocation, the runtime validates the operation against a policy store, identifying the invoking principal, the requested action, and the target resource. HADL is implemented in Rust via procedural macros with a four-phase compilation pipeline consisting of parsing, bidirectional type checking, code generation, and async tree-walking interpretation, and integrates with GitHub Copilot SDK for LLM execution.

We make the following contributions, reflecting a synthesis of programming language design, type theory, and agentic systems engineering.

\begin{enumerate}
	\item We design HADL, an imperative DSL in which \texttt{user}, \texttt{tool}, and \texttt{agent} are first-class typed expressions and workflows are curried functions, providing natural imperative syntax for multi-turn agent orchestration with mutable state, unbounded loops, structured iteration, and nested composition. Compile-time checking includes diagnostic error categories and source-span reporting, catching workflow errors before any LLM call is made.
	\item We adapt bidirectional type checking to the agentic domain, introducing a typing discipline in which LLM tool calls require binding-site type annotations, formalizing the boundary between typed program logic and untyped oracle outputs. We implement structural record subtyping with set-of-fields semantics, supporting width subtyping, depth subtyping, and field-order independence.
	\item We integrate Cedar-based authorization into the runtime evaluation loop, enforcing (principal, action, resource) triplets at each tool and agent invocation, embedding governance directly in the language's evaluation rules rather than treating it as a deployment-time concern.
	\item We present a complete implementation in Rust via procedural macros, comprising a parser, a bidirectional type checker, and an async tree-walking interpreter. We evaluate the system on case studies including code review, hierarchical planning, and iterative refinement.
\end{enumerate}
