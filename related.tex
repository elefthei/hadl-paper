\section{Related Work}
\label{sec:related}

\subsection{LLM Agent Frameworks}

The dominant approach to building agentic systems is through orchestration libraries. LangChain~\cite{langchain2022} provides composable abstractions for chaining LLM calls, with state-machine-style orchestration and human-in-the-loop interrupts layered on top. ReAct~\cite{yao2022react} establishes the interleaved-reasoning-and-acting pattern that HADL's \texttt{gen} and \texttt{agent} primitives formalize at the language level. AutoGen~\cite{wu2024autogen} introduces multi-agent conversation protocols, and MetaGPT~\cite{hong2023metagpt} assigns specialized roles to agents. None of these provides static type checking or per-invocation authorization. HADL treats agentic plans as typed programs with type checking and policy enforcement as language-level features. 
External reference-monitor systems such as PCAS~\cite{palumbo2026policy}, which compiles Datalog policies over a cross-agent dependency graph and blocks unauthorized tool calls before execution, pursue the same goal from outside the agent framework. HADL pursues it from inside, embedding \texttt{Policy} and \texttt{enforce} in the language.

Yue et al.~\cite{yue2026statictemplatesdynamicruntime} organize workflow optimization methods by when structure is determined and what representation is used, identifying expressivity versus verifiability as a central open problem. HADL supports dynamic structure determination through gradual validation while verifying everything statically known. HADL's \texttt{eval} is a typed realization of the pre-execution workflow generation pattern, and its self-healing specializes the verbal-feedback retry loop of Reflexion~\cite{shinn2024reflexion} to typed outputs: a structured type error, a Cedar denial, or a workflow-type mismatch becomes the signal that reruns the specific gen call that produced it. The survey does not address authorization, and HADL's Cedar-based enforcement fills this gap.

\subsection{Programming Language Approaches to LLM Interaction}

LMQL~\cite{beurer2023prompting} introduces a query language for LLMs with scripted prompting, constrained decoding, and control flow, representing the closest PL precedent. However, LMQL does not provide a type system for tool outputs or per-invocation authorization. DSPy~\cite{khattab2023dspy} optimizes LLM pipelines through automatic prompt tuning but operates at the pipeline level without a type system. Daunis~\cite{daunis2025declarative} proposes a declarative DSL using DAG-based definitions with compile-time validation but without unbounded iteration or a per-invocation authorization model. HADL combines static type checking with imperative control flow, mutable state, and integrated policy enforcement. Its \texttt{eval} expression draws on the metacircular interpretation tradition~\cite{mccarthy1960recursive,reynolds1972definitional}, applied here to LLM-generated workflow ASTs.

\subsection{Structured Generation and Typed LLM Interfaces}

A complementary line of work constrains LLM outputs to obey a schema at decoding time or validates them at parse time. Toolformer~\cite{schick2023toolformer} trains LLMs to invoke external tools through self-supervised interpolation of API calls, addressing the \emph{when to call} problem but without a type system for the returned values. SGLang~\cite{zheng2024sglang} is a structured-generation runtime that interleaves LLM inference with constrained decoding (finite-state automata over tokens) and program control flow; its guarantees live at the decoding layer, whereas HADL's live at the language layer and are enforced by a separate post-generation re-checker. TypeChat~\cite{typechat2023} is the closest practical precedent: it asks an LLM to produce JSON conforming to a TypeScript interface and retries with the compiler's error message on rejection. HADL generalises this pattern in three directions --- from value schemas to runtime-materialized \emph{types}, executable \emph{workflows}, and Cedar \emph{policies} --- and embeds it in a full language with mutable state, loops, per-invocation authorization, and a formal per-step soundness result. Guidance~\cite{guidance2023} is a template-based constrained-generation toolkit; it shapes single LLM outputs but does not address workflow-level typing or authorization. HADL is not a constrained-decoding system: the LLM is free to emit any string, and validity is checked after the fact against the annotated runtime type.

\subsection{Gradual Typing}

HADL's type system is inspired by gradual typing~\cite{siek2006gradual,siek2015refined}, which integrates static and dynamic typing through an unknown type compatible with all others. In HADL, \texttt{Schema} plays an analogous role, treated as universally compatible under the subtyping relation. HADL is not gradual typing in the strict sense: it inserts no runtime casts at type boundaries, does not satisfy the gradual guarantee~\cite{siek2015refined}, and does not claim the associated blame and refined-criteria results. Instead, when the LLM produces a concrete schema, the runtime pauses and inspects the program region extending from the current schema generation up to the next schema generation (or to the end of the program if none follows), collecting every field access on values of that schema's type and checking that each accessed field is actually declared by the schema. Validation is therefore scoped to the window in which the freshly arrived type is in force, rather than a re-run of the full type checker. The same mechanism extends to workflow and policy validation, where uses of the generated value (an \texttt{eval} call site, an \texttt{enforce} expression with downstream \texttt{as} clauses) are checked against what the LLM produced. The source of dynamic types is LLM execution rather than programmer choice, and we collectively term this discipline \emph{gradual validation}. We regard gradual validation as a design synthesis of staged re-checking plus bounded retry-based repair rather than as a new foundational type theory.
