\section{Related Work}
\label{sec:related}

\textbf{LLM agent frameworks.} The dominant approach is orchestration libraries: LangChain~\cite{langchain2022} chains LLM calls; ReAct~\cite{yao2022react} establishes the interleaved reasoning-acting pattern that \hadl{}'s \texttt{gen}/\texttt{agent} formalize at the language level; AutoGen~\cite{wu2024autogen} and MetaGPT~\cite{hong2023metagpt} add multi-agent protocols. None provides static type checking or per-invocation authorization. External reference monitors such as PCAS~\cite{palumbo2026policy} pursue the same safety goal from outside the framework; \hadl{} pursues it from inside. Yue et al.~\cite{yue2026statictemplatesdynamicruntime} identify expressivity vs.\ verifiability as a central open problem; \hadl{}'s \texttt{eval} is a typed realization of pre-execution workflow generation, and self-healing specializes the verbal-feedback retry loop of Reflexion~\cite{shinn2024reflexion} to typed outputs.

\textbf{Programming-language approaches to LLM interaction.} LMQL~\cite{beurer2023prompting} provides scripted prompting with constrained decoding but no type system for tool outputs or per-invocation authorization. DSPy~\cite{khattab2023dspy} optimizes pipelines through automatic prompt tuning at the pipeline level. Daunis~\cite{daunis2025declarative} is a DAG-based DSL with compile-time validation but no unbounded iteration or authorization model. \hadl combines static type checking with imperative control flow, mutable state, and integrated policy enforcement. Its \texttt{eval} draws on the metacircular interpretation tradition~\cite{mccarthy1960recursive,reynolds1972definitional} applied to LLM-generated ASTs.

\textbf{Structured generation and typed LLM interfaces.} Toolformer~\cite{schick2023toolformer} trains LLMs to invoke external tools without a type system. SGLang~\cite{zheng2024sglang} interleaves inference with constrained decoding (finite-state automata over tokens) at the decoding layer, whereas \hadl{}'s guarantees live at the language layer and are enforced by a post-generation re-checker. TypeChat~\cite{typechat2023} is the closest practical precedent: an LLM produces JSON conforming to a TypeScript interface and retries with the compiler's error on rejection. \hadl{} generalises this from value schemas to runtime-materialized \emph{types}, executable \emph{workflows}, and Cedar \emph{policies}, embedded in a language with mutable state, loops, and per-invocation authorization, with mechanized per-step soundness. Guidance~\cite{guidance2023} shapes single outputs via templates without workflow-level typing. \hadl{} is not a constrained-decoding system: the LLM emits any string and validity is checked post-hoc.

\textbf{Gradual typing.} \hadl{}'s type system is inspired by gradual typing~\cite{siek2006gradual,siek2015refined}: \texttt{Schema} plays the role of the unknown type. \hadl{} inserts no casts and does not claim the gradual guarantee or its blame results~\cite{siek2015refined}. When the LLM produces a concrete schema, the runtime re-checks the program region up to the next schema generation --- a window-scoped pass, not a full re-run --- and the same mechanism extends to workflows and policies. We term this discipline \emph{gradual validation}: staged re-checking plus bounded retry-based repair, not a new foundational type theory.
