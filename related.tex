\section{Related Work}
\label{sec:related}

\subsection{LLM Agent Frameworks}

The dominant approach to building agentic systems is through orchestration libraries. LangGraph~\cite{langgraph2024} models workflows as state machines with durable execution and human-in-the-loop interrupts, while LangChain~\cite{langchain2022} provides composable abstractions for chaining LLM calls. AutoGen~\cite{wu2024autogen} introduces multi-agent conversation protocols, and MetaGPT~\cite{hong2023metagpt} assigns specialized roles to agents. Durable execution platforms such as Temporal~\cite{temporal2024} provide fault-tolerant workflow infrastructure but without type systems for LLM interaction. None of these provides static type checking, compile-time diagnostics, or per-invocation authorization. HADL treats agentic plans as typed programs with static type checking and policy enforcement as language-level features.

Yue et al.~\cite{yue2026statictemplatesdynamicruntime} organize workflow optimization methods by when structure is determined and what representation is used, identifying expressivity versus verifiability as a central open problem. HADL supports dynamic structure determination through gradual generation while providing static verification of all statically known structure. HADL's \texttt{eval} is a typed realization of the pre-execution workflow generation pattern, and its self-healing implements the trace-derived feedback pattern the survey discusses. The survey does not address authorization; HADL's Cedar-based enforcement fills this gap.

\subsection{Programming Language Approaches to LLM Interaction}

LMQL~\cite{beurer2023prompting} introduces a query language for LLMs with scripted prompting and constrained decoding, representing the closest PL precedent, but targets single-turn prompt scripting without mutable state or unbounded loops. DSPy~\cite{khattab2023dspy} optimizes LLM pipelines through automatic prompt tuning but operates at the pipeline level without a type system. Daunis~\cite{daunis2025declarative} proposes a declarative DSL using DAG-based definitions but without types, imperative control flow, or unbounded iteration. HADL combines static type checking with imperative control flow, mutable state, and integrated policy enforcement.

\subsection{Gradual Typing}

HADL's type system is inspired by gradual typing~\cite{siek2006gradual,siek2015refined}, which integrates static and dynamic typing by introducing an unknown type compatible with all other types. In HADL, \texttt{Schema} plays an analogous role, where the subtyping relation treats \texttt{Schema} as universally compatible with all types (both as subtype and supertype), deferring actual type verification to runtime. However, HADL does not implement gradual typing in the standard sense. There are no runtime casts inserted at type boundaries, and HADL does not satisfy the gradual guarantee~\cite{siek2015refined} since adding \texttt{Schema} annotations triggers staged re-typechecking that can change runtime behavior. Instead, HADL implements \emph{staged re-typechecking}: when an LLM call materializes a concrete type, the type checker is re-invoked over the remaining program with the concrete type substituted, re-checking entire program regions rather than individual boundary crossings. The source of dynamic types is LLM execution rather than programmer choice, and the same staged mechanism extends to workflow generation and policy generation, which we collectively term \emph{gradual generation}.

Tobin-Hochstadt and Felleisen~\cite{tobinhochstadt2006interlanguage,tobinhochstadt2008design} developed Typed Racket with sound interoperation between typed and untyped code. Vitousek et al.~\cite{vitousek2017big} addressed type soundness in open-world settings. For structural typing, HADL follows Cardelli's~\cite{cardelli1984semantics} record types and R\'{e}my's~\cite{remy1989type} field-order independence, with Pierce~\cite{pierce2002types} and Cardelli and Wegner~\cite{cardelli1985understanding} providing type-theoretic foundations.

\subsection{Metacircular Interpretation and Policy Languages}

HADL's \texttt{eval} expression draws on the metacircular interpretation tradition~\cite{mccarthy1960recursive,reynolds1972definitional,landin1964mechanical,abelson2022structure,friedman2001essentials,amin2017collapsing}, where LLM tool calls generate HADL workflows as JSON-serialized ASTs, and the typed \texttt{eval} expression deserializes and executes them with type checking at the call site.

HADL integrates Cedar~\cite{cutler2024cedar}, a formally verified policy language supporting (principal, action, resource) authorization. HADL's contribution is embedding Cedar's model directly into the runtime evaluation loop and enabling gradual policy generation where LLMs produce Cedar policies at runtime that are compiled and merged in an append-only fashion.
