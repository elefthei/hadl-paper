\section{Related Work}
\label{sec:related}

\textbf{LLM agent frameworks.} The dominant approach to building agentic systems is orchestration libraries. LangChain~\cite{langchain2022} chains LLM calls with state-machine-style orchestration. ReAct~\cite{yao2022react} establishes the interleaved reasoning-acting pattern that \hadl's \texttt{gen}/\texttt{agent} formalize at the language level. AutoGen~\cite{wu2024autogen} and MetaGPT~\cite{hong2023metagpt} add multi-agent protocols and role specialization. None provides static type checking or per-invocation authorization. External reference-monitor systems such as PCAS~\cite{palumbo2026policy}, which compiles Datalog policies over a cross-agent dependency graph, pursue the same safety goal from outside the agent framework. \hadl pursues it from inside, embedding \texttt{Policy} and \texttt{enforce} in the language. Yue et al.~\cite{yue2026statictemplatesdynamicruntime} organize workflow optimization methods by when structure is determined, identifying expressivity versus verifiability as a central open problem. \hadl's \texttt{eval} is a typed realization of the pre-execution workflow-generation pattern, and its self-healing specializes the verbal-feedback retry loop of Reflexion~\cite{shinn2024reflexion} to typed outputs.

\textbf{Programming-language approaches to LLM interaction.} LMQL~\cite{beurer2023prompting} provides scripted prompting with constrained decoding but no type system for tool outputs or per-invocation authorization. DSPy~\cite{khattab2023dspy} optimizes pipelines through automatic prompt tuning at the pipeline level. Daunis~\cite{daunis2025declarative} is a DAG-based DSL with compile-time validation but no unbounded iteration or authorization model. \hadl combines static type checking with imperative control flow, mutable state, and integrated policy enforcement. Its \texttt{eval} draws on the metacircular interpretation tradition~\cite{mccarthy1960recursive,reynolds1972definitional} applied to LLM-generated ASTs.

\textbf{Structured generation and typed LLM interfaces.} Toolformer~\cite{schick2023toolformer} trains LLMs to invoke external tools but without a type system for returned values. SGLang~\cite{zheng2024sglang} interleaves inference with constrained decoding (finite-state automata over tokens). Its guarantees live at the decoding layer, whereas \hadl's live at the language layer and are enforced by a post-generation re-checker. TypeChat~\cite{typechat2023} is the closest practical precedent: it asks an LLM to produce JSON conforming to a TypeScript interface and retries with the compiler's error on rejection. \hadl generalises this pattern from value schemas to runtime-materialized \emph{types}, executable \emph{workflows}, and Cedar \emph{policies}, embedding it in a full language with mutable state, loops, per-invocation authorization, and a mechanized per-step soundness result. Guidance~\cite{guidance2023} shapes single LLM outputs via templates but does not address workflow-level typing or authorization. \hadl is not a constrained-decoding system: the LLM is free to emit any string, and validity is checked after the fact.

\textbf{Gradual typing.} \hadl's type system is inspired by gradual typing~\cite{siek2006gradual,siek2015refined}: \texttt{Schema} plays a role analogous to the unknown type, treated as universally compatible. \hadl inserts no casts, does not satisfy the gradual guarantee~\cite{siek2015refined}, and does not claim the associated blame results. When the LLM produces a concrete schema, the runtime re-checks the program region up to the next schema generation, a window-scoped pass rather than a full re-run. The same mechanism extends to workflow and policy validation. We term this discipline \emph{gradual validation}: a design synthesis of staged re-checking plus bounded retry-based repair, not a new foundational type theory.
