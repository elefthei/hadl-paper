\section{Related Work}
\label{sec:related}

\textbf{LLM agent frameworks.} The dominant approach is orchestration libraries: LangChain~\cite{langchain2022} chains LLM calls; ReAct~\cite{yao2022react} establishes the interleaved reasoning-acting pattern that \pact{}'s \texttt{gen}/\texttt{agent} formalize at the language level; AutoGen~\cite{wu2024autogen} and MetaGPT~\cite{hong2023metagpt} add multi-agent protocols. None provides static type checking or per-invocation authorization. External reference monitors such as PCAS~\cite{palumbo2026policy} pursue the same safety goal from outside the framework; \pact{} pursues it from inside. Yue et al.~\cite{yue2026statictemplatesdynamicruntime} identify expressivity vs.\ verifiability as a central open problem; \pact{}'s \texttt{eval} is a typed realization of pre-execution workflow generation, and self-healing specializes the verbal-feedback retry loop of Reflexion~\cite{shinn2024reflexion} to typed outputs.

\textbf{Programming-language approaches to LLM interaction.} LMQL~\cite{beurer2023prompting} provides scripted prompting with constrained decoding but no type system for tool outputs or per-invocation authorization. DSPy~\cite{khattab2023dspy} optimizes pipelines through automatic prompt tuning at the pipeline level. Daunis~\cite{daunis2025declarative} is a DAG-based DSL with compile-time validation but no unbounded iteration or authorization model. \pact combines static type checking with imperative control flow, mutable state, and integrated policy enforcement. Its \texttt{eval} draws on the metacircular interpretation tradition~\cite{mccarthy1960recursive,reynolds1972definitional} applied to LLM-generated ASTs.

\begin{sloppypar}
\textbf{Structured generation and typed LLM interfaces.} Toolformer~\cite{schick2023toolformer} trains LLMs to invoke external tools without a type system. SGLang~\cite{zheng2024sglang} interleaves inference with constrained decoding (finite-state automata over tokens) at the decoding layer, whereas \pact{}'s guarantees live at the language layer and are enforced by a post-generation re-checker. TypeChat~\cite{typechat2023} is the closest practical precedent: an LLM produces JSON conforming to a TypeScript interface and retries with the compiler's error on rejection. \pact{} generalises this from value schemas to runtime-materialized \emph{types}, executable \emph{workflows}, and Cedar \emph{policies}, embedded in a language with mutable state, loops, and per-invocation authorization, with mechanized per-step soundness. Guidance~\cite{guidance2023} shapes single outputs via templates without workflow-level typing. \pact{} is not a constrained-decoding system: the LLM emits any string and validity is checked post-hoc.
\end{sloppypar}

\textbf{Gradual typing.} \pact{}'s type system is inspired by gradual typing~\cite{siek2006gradual,siek2015refined}: \texttt{Schema} plays the role of the unknown type. \pact{} inserts no casts and does not claim the gradual guarantee or its blame results~\cite{siek2015refined}. When the LLM produces a concrete schema, the runtime re-checks the program region up to the next schema generation --- a window-scoped pass, not a full re-run --- and the same mechanism extends to workflows and policies. We term this discipline \emph{gradual validation}: staged re-checking plus bounded retry-based repair, not a new foundational type theory.

\textbf{Contracts, refinement, and authorization DSLs.} The post-generation re-checker resembles a higher-order contract~\cite{findler2002contracts}: an LLM-produced workflow or policy is wrapped at the boundary where it is consumed, and a violation diverts control to the producer (the causal \texttt{gen}) rather than blaming the caller. Refinement-type systems such as LiquidHaskell~\cite{vazou2014liquidhaskell} discharge predicates statically, whereas \pact{} validates dynamically against artifacts that did not exist at compile time, including types and policies. On the authorization side, Cedar~\cite{cutler2024cedar}, OPA/Rego~\cite{opa2024}, and IAM-style systems share the role of an external policy decision point; \pact{} differs in treating policies as first-class, generated language data subject to the same install-only-shrinks discipline as the rest of the workflow.
