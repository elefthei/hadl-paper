\section{Formal Model}
\label{sec:semantics}

This section gives an operational semantics for \pact that interleaves
evaluation, type checking, and policy checking, with self-healing on
recoverable failures. Types, workflows, and authorization policies
materialize at runtime, and self-heal steps deliberately revisit
configurations whose continuation does not yet type-check. The semantics
therefore (i) is parameterized by a non-deterministic LLM oracle, (ii)
uses substitution-based reduction so that materialized values flow into
their continuations by $\beta$-substitution, and (iii) states soundness
relative to an oracle that eventually returns a well-typed authorized
response within the declared retry budget.

\subsection{Machine Configuration and Judgments}
\label{sec:sem-config}

A \pact configuration is a four-tuple
$C = \MConfigS{\MErr}{\MPol}{\MStore}{e}$. The component $\MErr$ is the
\emph{heal context}: a list of failed-attempt feedback events
$\ErrEv{\epsilon}$ whose length $|\MErr|$ is the current retry count.
A successful oracle step flushes $\MErr$ to $\emptyset$, so retries are
local to the current materialization site and old feedback never affects
later oracle calls. The component $\MPol$ is the installed Cedar policy,
updated only by $\Enforce{v}$ via an append-only merge that can only
shrink the allow set --- this invariant is the basis of policy
monotonicity (T3). The component $\MStore$ is a finite map from
mutable-variable names to $(\tau, v)$ pairs, mutated only by
\texttt{var} declarations and assignments. The component $e$ is the
expression under reduction.

The two binding mechanisms in \pact are independent: \texttt{let},
closure application, and $\rulename{For-Cons}$ use capture-avoiding
substitution, while \texttt{var} declarations, assignments, and reads
use a separate name-store. Rules that leave $\MStore$ unchanged elide it
and use the three-component notation $\MConfig{\MErr}{\MPol}{e}$; rules
that read or write the store are collected in \sect{app:rules-mut}. Each
oracle-facing expression carries its calling principal explicitly
($\Gen{\tau}{s}{\MPrin}$, $\Agent{s}{\MPrin}$, and closure-application
syntax) so authorization can be checked at the invocation site. Values
are the subset satisfying $\mathsf{isValue}$ (base literals, records,
arrays, schema values $\Tschema(\tau)$, Cedar policy values, closures).

Three judgments drive correctness. \emph{Runtime typing}
$\StType{v}{\tau}$ holds of a closed value $v$ against type $\tau$; it
is the closed-value, $\mathsf{ok}$ branch of the more general
typing judgment $\Gamma \vdash t : \tau \Rightarrow r$ with
$r \in \{\mathsf{ok}, \mathsf{err}(\epsilon)\}$. After substitution, the
runtime invokes the same type checker used statically on the remaining
program; the proofs use only its $\mathsf{ok}$/$\mathsf{err}$ outcomes,
not its internal algorithm. \emph{Static principal admissibility}
$\PrinDeclared{\MPol}{\MPrin}$ holds when the principal $\MPrin$ named
in an \texttt{as} clause is declared in the current policy state, and
is the only authorization premise enforced statically at every
\texttt{gen}/\texttt{agent} redex; if the redex omits an \texttt{as}
clause, $\MPrin$ defaults to \texttt{root}, which is always declared.
The action and resource of any specific access span the filesystem
(\texttt{Read}/\texttt{Write}/\texttt{Exec} on files and directories)
and are decided just-in-time by the runtime monitor of
\sect{sec:implementation}, not at the redex. The runtime monitor sits
outside the mechanized core: the formal development verifies the static admissibility
premise and the install discipline; the action/resource decision at
each filesystem access is enforced by the prototype runtime only.
\emph{Well-formedness} $\WF{C}$ requires both $|\MErr| \le \MRetry$ and
that every store cell is well-typed:
$\forall~x,\tau,v.\; \MStore(x) = (\tau, v) \Rightarrow \StType{v}{\tau}$.

\subsection{Reduction}
\label{sec:sem-reduction}

The reduction relation $C \step C'$ is a single small-step relation
parameterized by the oracle $\mathcal{O}$. \emph{Root-level} rules fire
when the top of $e$ matches; \emph{structural congruence} rules drive
reduction to the leftmost subterm under CBV. Two root rules perform
substitution directly: $\rulename{Let-Bind}$ rewrites $\Let{\tau}{v}{e}$
to $\Subst{e}{v}$ (capture-avoiding substitution of $v$ for $x$ in $e$),
and $\rulename{Eval-Success}$ performs the analogous simultaneous
substitution into a closure body.

Self-healing is \emph{let-centric} and \emph{continuation-driven}.
$\Gen{\tau}{s}{\MPrin}$ reduces only as the immediate right-hand side of
a $\texttt{let}$, so each materialization can validate the oracle
response against its continuation before committing.

We classify self-healing opportunities by the predicate $\Healable{\tau}$,
defined inductively: $\mathsf{true}$ on Schema, Policy, and arrow types,
and on records or arrays containing a healable component;
$\mathsf{false}$ on the base scalars Unit, Bool, Number, String.
Intuitively, schemas, policies, workflows, and containers that include
them can change how the continuation type-checks, so they participate
in retry; scalars commit on success or fail closed without retry.

At each healable $\tau$ the runtime performs two checks: the oracle
response $v$ must satisfy $\StType{v}{\tau}$, and the continuation $p$
must type-check under $\Gext{x}{\tau}$. A well-typed $v$ is a
prerequisite for the heal rules; self-healing is driven by
\emph{continuation-typing} failure, whose error string $\epsilon$ is
appended to $\MErr$ and fed back to the oracle. Continuation checks use
$\TJok{\Gext{x}{\tau}}{p}{\tau'}$ on success and
$\TJerr{\Gext{x}{\tau}}{p}{\tau'}{\epsilon}$ on failure;
$\StType{v}{\tau}$ is the closed-value specialization of the $\mathsf{ok}$
case (Appendix~\ref{app:rules} lists all rules).

Each healable $\tau$ is handled by the parametric pair
$\rulename{Healable-Ok}$ and $\rulename{Healable-Heal}$, with
$\rulename{Schema-Ok}$, $\rulename{Schema-Heal}$ and
$\rulename{Arrow-Ok}$, $\rulename{Arrow-Heal}$ as the named Schema and
Arrow specializations. Two further rules apply at gen sites: at
non-healable $\tau$, $\rulename{Gen-TypeErr}$ steps an ill-typed
response to the fail-closed sink $\errV$; at healable $\tau$,
$\rulename{Gen-Budget}$ steps to $\errV$ when $|\MErr| > \MRetry$. The
shared rule $\rulename{Gen-PolHeal}$ records a feedback event and
retries when the principal is denied
($\lnot\PolAllow{\MPol}{\MPrin}{\texttt{gen}}$).

\figr{fig:rules-core} shows the Schema instance of the
continuation-driven pattern. The arrow case is structurally identical
with $\tau = \Tarrow{T_1 \to \cdots \to T_n}{T'}$; both are instances of
$\rulename{Healable-Ok}$, $\rulename{Healable-Heal}$, which also covers
the Policy case ($\Healable{\Tpolicy} = \mathsf{true}$) and nested
compound healable types such as $\Tarr{\Tschema}$ from the
clinical-trial \texttt{Array[crf]} pattern (Appendix~\ref{app:rules}).
For arrows, $\Subst{p}{v}$ substitutes the oracle-returned closure into
the continuation, mechanized via capture-avoiding substitution.

T2 (\sect{thm:pact-soundness}) guarantees that every committed value
other than the distinguished fail-closed sink $\errV$ carries a runtime
type --- $\errV$ is intentionally the only inhabitant lacking an
$\StType{}{}$ derivation. The standalone $\Agent{s}{\MPrin}$ has its
own pair $\rulename{Agent-Success}$, $\rulename{Agent-PolHeal}$:
$\Agent{s}{\MPrin}$ materializes $\Tstring$, so it follows the
non-healable path of returning a checked string on success and retrying
on principal denial.

\begin{sloppypar}
The pure-core rules ($\rulename{If-True/False}$, $\rulename{While}$,
$\rulename{For-Nil/Cons}$, $\rulename{Seq}$,
$\rulename{Ask}$, $\rulename{Say}$, $\rulename{Enforce-Install}$) follow
ordinary left-to-right CBV evaluation; $\rulename{For-Cons}$ uses
substitution to bind the iteration variable. The only rule that updates
$\MPol$ is $\rulename{Enforce-Install}$, whose install operation rejects
blanket $\texttt{forbid}$ under a deny-all guard, rejects unknown action
types, and merges rules append-only. These checks ensure that
installation can only shrink the allow set
($\Allow{\MPol'} \subseteq \Allow{\MPol}$), yielding T3.
\end{sloppypar}

\subsection{Correctness}
\label{sec:sem-correctness}

We state safety in a staged form suited to \pact's runtime
materialization: per-step preservation of well-formedness, runtime
typeability of committed non-sink values, policy monotonicity, and
progress for oracle redexes under oracle assumptions. All four
properties are parameterized by the oracle; trace-level termination is
argued on paper.

\begin{definition}[Eventually-truthful oracle]
\label{def:oracle-truthful}
An oracle $\mathcal{O}$ is \emph{eventually-truthful at budget $\MRetry$}
for a site $(s,\tau)$ under an authorization predicate
$A : V \to \mathrm{Prop}$ if there exists a heal context $\MErr$ with
$|\MErr| \le \MRetry$ and a value $v$ satisfying
$\Oracle{s}{\MErr}{\tau} \ni v$, $\StType{v}{\tau}$, and $A(v)$. The
main theorems instantiate $A(v) \equiv \top$; the parameter lets
specialized sites add semantic admissibility conditions without
weakening the progress statement. The assumption does not require the
oracle to answer correctly on every attempt; it requires only that some
feedback history within budget yields a well-typed authorized response.
\end{definition}

\begin{sloppypar}
This matches the existential shape of the oracle predicate: it witnesses some well-typed
response at \emph{some} heal context, not necessarily the caller's
current $\MErr$.
\end{sloppypar}

\paragraph{What soundness means here.} The four claims below are one
staged guarantee, parameterized by the configuration's heal context
$\MErr$, policy state $\MPol$, and store $\MStore$. $\MErr$ is the
per-site obligation cache: a successful materialization flushes it to
$\emptyset$, so retries do not leak across sites; a heal step appends
exactly one new feedback event, giving the next oracle attempt the
precise reason the previous one failed. The retry budget $\MRetry$ caps
the number of heal events before the configuration sinks to $\errV$ ---
this is the operational meaning of \emph{fail-closed}, and it is the
only way a non-value step can terminate without producing a well-typed
result. The healable predicate $\Healable{\tau}$ is exactly the
inductive class of types for which retry is admissible: base scalars
commit immediately, while schemas, policies, arrows, and their
recursive containers may re-enter the oracle loop until the
continuation type-checks or the budget is spent. Together, T1 says the
budget and store invariants survive every step, T2 says every committed
non-sink value carries a runtime type, T3 says authority shrinks
monotonically, and T4 says an oracle redex either commits, heals, or
fails closed --- never gets stuck. All four are mechanized.

\begin{theorem}[Soundness]
\label{thm:pact-soundness}
For oracle $\mathcal{O}$ and step $C \step C'$:
\begin{enumerate}
  \item[\textbf{T1}] (WF-preservation) if $~\WF{C}$ then $\WF{C'}$.
  \item[\textbf{T2}] (non-sink typeability) if $~C' = \MConfigS{\MErr'}{\MPol'}{\MStore'}{v}$ with $\mathsf{isValue}(v)$ and $v \neq \errV$, then $\exists~\tau.\ \StType{v}{\tau}$.
  \item[\textbf{T3}] (policy monotonicity) $~\Allow{\MPol'} \subseteq \Allow{\MPol}$.
\end{enumerate}
\end{theorem}

T2 produces an $\StType{v}{\tau}$ witness for every committed value
other than the fail-closed sink $\errV$ (Appendix~\ref{app:soundness}
expands the per-rule cases).

\begin{theorem}[Progress for oracle redexes]
\label{thm:pact-progress}
{\sloppy Let $C = \MConfigS{\MErr}{\MPol}{\MStore}{e}$ with $\WF{C}$.
\textbf{T4a} (standalone-gen non-redex): if $e = \Gen{\tau}{s}{\MPrin}$,
then $C$ has no successor; materialization happens only inside a
surrounding $\Let{\tau}{\Gen{\tau}{s}{\MPrin}}{p}$.
\textbf{T4b} (truthful Schema success): if
$e = \Let{\Tschema}{\Gen{\Tschema}{s}{\MPrin}}{p}$,
$\PolAllow{\MPol}{\MPrin}{\texttt{gen}}$, the oracle returns a
well-typed Schema response at the current $\MErr$, and
$\TJok{\Gext{x}{\Tschema}}{p}{\tau'}$, then $\exists~C'.\ C \step C'$
via $\rulename{Schema-Ok}$.
\textbf{T4c} (parametric progress for healable gen and agent): with the
same authorization hypothesis, a direct oracle witness at the caller's
$\MErr$, and the corresponding continuation-typing premise (for gen) or
$\StType{v}{\Tstring}$ (for agent), $\exists~C'.\ C \step C'$.\par}
\end{theorem}

T4b and T4c are instances of the parametric Healable progress lemma;
the appendix gives the corresponding Arrow, Policy, and nested-healable
corollaries together with the trace-level termination argument
(Appendix~\ref{app:soundness}, \ref{app:term}).

\begin{sloppypar}
\noindent\textbf{Scope.} T1--T4c are mechanized and depend only on
standard meta-axioms.
Trace-level termination is prose-only. Progress conclusions assume an
eventually-truthful oracle; without that assumption T1--T3 still hold
per step, and oracle redexes either reach $\errV$ via the explicit sink
rules or halt at the redex. The threat-model gaps of
\sect{sec:design-threat} are outside the scope of soundness; practical
claims are the subject of \sect{sec:evaluation}.
\end{sloppypar}
